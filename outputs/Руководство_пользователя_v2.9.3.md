# Руководство пользователя

**ProductCamp Conversion Analytics Tool**
Версия 2.9.3 · Май 2026

> Локальный аналитический инструмент для трека «Конверсии и лидген» ProductCamp:
> вытаскивает данные Яндекс.Метрики, строит интерактивный дашборд по конверсиям
> и генерирует отчёт с проверяемыми гипотезами по росту платных продаж.
>
> Репозиторий: `git@github.com:Fighter90/yandex-metrika-analyse-dashboard.git`

---

## 1. Что это и зачем

Инструмент отвечает на один вопрос: **откуда берутся платные билеты и как добрать до 300**. Он не заменяет Яндекс.Метрику, а превращает её сырые цели в понятную картину: разбивку по каналам, воронку «визит → заявка → оплата», приоритизированные гипотезы роста и Decision Log.

**Два сквозных принципа:**

- **Заявка ≠ оплата.** Достижение цели в Метрике (заявка) и фактическая оплата разделены везде в интерфейсе и в отчёте. Прогресс к KPI считается по оплатам.
- **Anti-hallucination.** Любое число на дашборде и в отчёте прослеживается до сырого ответа Метрики в таблице `raw_responses` (SQLite). Выдуманных цифр нет — это инвариант.

## 2. Быстрый запуск

Подробная инструкция установки для Windows и macOS — в отдельном документе **«Установка ProductCamp Analytics (Windows + macOS) v2.9.3»**.

Самый короткий путь:

```bash
git clone git@github.com:Fighter90/yandex-metrika-analyse-dashboard.git
cd yandex-metrika-analyse-dashboard
./setup.sh          # install → init → start
```

После старта откроется:

- **Дашборд:** http://localhost:5173
- **API:** http://localhost:4000
- **Swagger:** http://localhost:4000/docs

Без OAuth-токена дашборд поднимется на демо-данных (`pnpm seed`).

## 4. Фильтры

Глобальная панель в шапке (кроме Настроек и Справки):

- **Пресеты:** 7д, 14д, 30д, 90д, 1г.
- **Кастомный период:** кнопка «📅 Даты» → от/до → «Применить» (максимум 365 дней).
- **Сегмент:** B2C / B2C+B2B (по умолчанию) / B2B.
- **Архивные цели:** чекбокс — добавляет в выборку архивные цели Метрики.
- **Мобайл (< 768px):** кнопка «Фильтры» → bottom-sheet.

Изменение фильтра автоматически обновляет данные на всех страницах.

## 5. Генерация отчёта (`/report`)

1. **«Сформировать срез данных»** — собирает неизменяемый снимок (`snapshotId`). KPI «ОПЛАТ» (при e_purchase), воронка, B2B-пайплайн, превью 17 разделов.
2. **«Сгенерировать AI-анализ»** _(нужен `ANTHROPIC_API_KEY`)_ — 6 секций нарратива: **Итог · Каналы UTM Аудитория · Страницы Воронка · Риски Рекомендации · Гипотезы Дорожная карта · Итоговый вывод**. После генерации кнопка блокируется («AI-анализ сгенерирован»).
3. **«Export DOCX»** — файл по ГОСТ Р 7.32-2017 (Times New Roman 14pt, межстрочный 1.5, поля 30/15/20/20 мм, нумерация страниц, ГОСТ-титул, «Содержание», 3 PNG-графика, выравнивание по ширине).
4. **«Export PDF»** — PDF той же структуры через локальный Chrome.
5. **«Перестроить отчёт»** — пересборка снапшота с актуальными данными.

**Детерминизм:** один `snapshotId` → идентичный контент DOCX/PDF (PNG-имена = SHA1 содержимого). Файловый SHA отличается из-за zip-timestamp — это спецификация, не дефект.

## 6. KPI и согласованность

KPI к цели **300 платных билетов** считается единообразно на Обзоре, Целях и в снапшоте:

```
gap = target − (b2cApplications + b2bPaidTickets)
```

- При цели-покупке (e_purchase) `b2cApplications` = реальные оплаты Метрики → метка «**ОПЛАТ**».
- При цели-заявке `b2cApplications` = заявки, с пометкой «заявка ≠ оплата».
- B2B оплаты — ручной ввод в `b2b_manual` через **Настройки → B2B-пайплайн**.

Числа «Gap до цели», «N оплачено из 300», ring 17.3%, «Осталось 248» — побайтово совпадают на всех страницах и в DOCX/PDF.

## 7. Гипотезы и Decision Log

Гипотезы формулируются по методологии проверки гипотез:

```
{ЦА} {готова сделать / готова заплатить за / переключится на} {решение}, если {условие}.

+ ≥3 допущения (по 3 категориям: ценность, поведение, технология)
+ ≥2 метода проверки
+ светофор 🟢/🟡/🔴 с конкретной метрикой
+ дедлайн
+ ICE = Impact × Confidence × Ease (произведение, 1–1000)
```

UI блокирует сохранение гипотезы без полного формата. Decision Log замыкает цикл: каждый завершённый цикл проверки заканчивается записью в `decisions` с привязкой к данным.

## 8. Настройки

| Поле                  | Что делает                                                                                  |
| --------------------- | ------------------------------------------------------------------------------------------- |
| OAuth-токен           | Yandex Metrika scope `metrika:read`. Маска `y0__****DE`. Пустое значение не перезаписывает. |
| Client ID / Secret    | Для OAuth flow `pnpm auth`.                                                                 |
| COUNTER_ID            | ID счётчика Яндекс.Метрики. Не маскируется (не секрет).                                     |
| **GOAL_ID combobox**  | Поиск по 129 активным + 129 архивным целям. 0 = авто-определение (выбирает оплату/покупку). |
| ANTHROPIC_API_KEY     | Для AI-анализа. Маска `sk-a****AA` под полем. Пустое значение не перезаписывает.            |
| «🔄 Обновить данные»  | Re-sync из Метрики с прогресс-баром (10 стадий). После — refetch всех страниц.              |
| B2B-пайплайн collapse | CRUD сделок: компания, билеты, этап, сумма, дата.                                           |

**Кнопка «Сохранить»** обновляет только переданные ключи в `.env`, сохраняя остальные строки и комментарии. Маскированные секреты НЕ затираются.

В v2.9.3 в DOCX/PDF:

- Нет пустых разделов «Нет данных» / «Гипотезы не заведены» / «Завершённых проверок нет»
- Нет двойной нумерации в TOC
- AI-заголовки без эмодзи в начале
- Markdown/HTML-теги (`####`, `<p>`, `<br>`, `<strong>`, `<em>`) полностью очищены
- Размер ~ 160 КБ (без AI) / ~ 200 КБ (с AI)
- 3 PNG-графика встроены с SHA1-именами

## 11. Типичные сценарии

**Сценарий 1: «Какие каналы дают платных пользователей?»**

- Обзор → Микс каналов / UTM-таблица → Слабые места → клик «→ гипотеза» по каналу с трафиком и низким CR.

**Сценарий 2: «Сколько до цели?»**

- Цели → ring «N/300» → «Что если»-симулятор: подвигать «+% трафика» и «+% CR», увидеть итоговую оценку оплат и Gap.

**Сценарий 3: «Готовлю отчёт для руководства»**

- Отчёт → «Сформировать срез данных» → «Сгенерировать AI-анализ» → «Export DOCX» → отправить.

**Сценарий 4: «Проверяю гипотезу»**

- Отчёт → найти гипотезу в списке → провести проверку → закрыть в Decision Log с привязкой к данным.

**Сценарий 5: «Обновить данные за последнюю неделю»**

- Настройки → «🔄 Обновить данные из Метрики» → подождать прогресс-бар → дашборд обновлён.

## 12. FAQ

**Почему KPI «ОПЛАТ», а не «Заявки B2C»?**
Цель счётчика — `e_purchase` (Ecommerce: покупка), её достижения = реальные оплаты. Если поставите цель-заявку через GOAL_ID, метка станет «Заявки B2C» с пометкой «заявка ≠ оплата».

**Где увидеть, какой канал реально приносит деньги?**
Воронка → CR по каналам; Обзор → Слабые места; Отчёт → секция «Анализ по каналам трафика».

**Почему страницы выхода пустые?**
Метрика API часто не возвращает `exitURL` для счётчиков с фокусом на e-commerce. Это нормально — раздел в отчёте автоматически скрывается.

**UTM-coverage 36%, это плохо?**
≥ 70% — норма; < 70% — часть трафика без UTM выпадает из атрибуции. Размечайте кампании.

**Можно использовать без AI-ключа?**
Да. Без `ANTHROPIC_API_KEY` дашборд и отчёт работают, отсутствует только нарратив (6 AI-секций).

**Где хранятся данные?**
Локально, в SQLite `data/productcamp.sqlite`. Облака нет. `.env` с токенами — в `.gitignore`.

## 13. Глоссарий

- **Заявка** — достижение цели Метрики (например, отправка формы). Не равна оплате.
- **Оплата (B2C)** — достижение цели-покупки (e_purchase) — это фактическая оплата.
- **Оплата (B2B)** — оплаченный билет из ручного пайплайна `b2b_manual`.
- **CR (conversion rate)** — доля визитов, дошедших до цели (reaches / visits).
- **Bounce rate** — доля визитов с отказом (один экран, без действия).
- **ICE** — Impact × Confidence × Ease (произведение, 1–1000) — балл приоритета гипотезы.
- **Double Diamond** — Define (проблемы) → Develop (решения) → Deliver (решения зафиксированы).
- **Светофор 🟢/🟡/🔴** — пороги исхода проверки гипотезы с конкретной метрикой.
- **raw_responses** — таблица сырых ответов Метрики в SQLite; источник правды для всех цифр.
- **ИД среза (snapshotId)** — идентификатор неизменяемого среза данных; гарантирует идентичность DOCX/PDF/экрана.
- **UTM-coverage** — доля визитов с UTM-метками относительно всех визитов; считается из `utm_stats`.
- **WoW** — Week-over-Week, сравнение последних 7 дней с предыдущими 7.
