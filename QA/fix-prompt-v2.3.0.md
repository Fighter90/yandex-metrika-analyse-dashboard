## 5. Дополнительные графики, данные, таблицы, отчёты, гипотезы (мозговой штурм по запросу пользователя)

> Каждый пункт оформить как фичу: спека в `docs/specs/NNN-*.md` → tests → impl. Приоритет: high/med/low.

### 5.1. Графики, которых сейчас нет (high)

1. **Cohort retention по дате визита** — heatmap (X = дата визита, Y = дни после визита, цвет = % вернувшихся). Поможет понять, успевает ли пользователь принять решение об оплате.
2. **Sankey-диаграмма «UTM-source → landing → goal/no-goal»** — наглядно: какой источник попадает на какую страницу и кто конвертит.
3. **Heat-map по времени суток × дню недели** (визиты, заявки, CR). Помогает планировать рекламные посты.
4. **Funnel by channel** — стэк-бар: на каждом этапе воронки доля каждого канала.
5. **Top-N landing pages по микро-конверсиям**: scroll-depth, time-on-page, share-events (если включён GA enhanced или Метрика events).
6. **Time-to-payment distribution** — гистограмма «дней от первого визита до оплаты» (нужен `client_id` join `b2b_manual`).
7. **B2B pipeline velocity** — Sankey с этапами (Lead → Qualified → Negotiation → Closed Won/Lost), bottlenecks подсвечиваются.
8. **WoW / MoM сравнения** — на каждом ключевом графике добавить «предыдущий период» пунктиром.

### 5.2. Новые таблицы (med)

1. **Channel ROI matrix** — channel × cost (если введёшь рекламные затраты вручную) × visits × conv × revenue per conv → CAC, LTV.
2. **UTM-кампания × День** pivot — для поиска «прогоревших» кампаний.
3. **Landing pages × Channel** pivot — какой канал куда приводит трафик и где CR.
4. **Sources of truth** таблица: на каждую цифру в отчёте — источник (`raw_responses.id`, query_hash, attribution). Обязательно для anti-hallucination и аудита.
5. **Hypothesis × Snapshot × Decision** табличная связка — для трассировки «гипотеза → данные → решение».

### 5.3. Новые отчёты / разделы (high)

1. **Weekly digest** — раз в неделю автогенерируется снапшот + краткий PDF (1 страница) для рассылки команде.
2. **Анти-галлюцинация Audit** — отдельный отчёт «trace-by-trace», где каждый KPI в отчёте подписан query_hash и timestamp Метрики.
3. **What-if симулятор** — слайдеры: «увеличить CR mailing с 0% до 1%», пересчитать «сколько оплат добавит при текущем трафике». ECharts + live recompute.

### 5.4. Гипотезы, которые **данные уже сейчас** провоцируют

> Формат: `{ЦА} {готова Х} если {условие}`. Минимум 3 допущения, 2 метода, светофор, дедлайн, ICE=I×C×E.

#### H-001 (problem, problem-hypothesis)

> Целевая аудитория ProductCamp (продакты, маркетологи РФ) **не доходит до оплаты с Direct-трафика** (37 649 визитов, CR 0.5%, оплат 196 за 14 дней), потому что landing говорит про конференцию, но не про оплату.

- Допущения: (1) Direct = люди, которые уже знают бренд; (2) у них нет блокеров узнавания; (3) проблема — на странице решения.
- Методы проверки: (a) кастдев 5 человек, кто посетил, но не купил; (b) heatmap (Метрика) — где нажимают вместо CTA.
- Светофор: 🟢 если 3+ респондента подтвердят blocker «не нашёл цену» / «не понял, что надо платить».
- Дедлайн: за 1 неделю до старта кемпа.
- ICE: I=4, C=3, E=4 → **48**.

#### H-002 (solution, solution-hypothesis)

> Команда ProductCamp **готова добавить sticky-CTA «Купить билет» на странице productcamp.ru/summer2026**, если на 1-неделе A/B-теста CR Direct вырастет с 0.5% до 1.5% (+200%).

- Допущения: (1) sticky-CTA не сломает дизайн; (2) текущие 196 оплат не заберут просто перераспределение; (3) Метрика покажет статзначимость на 14 дней.
- Методы: A/B-тест через Метрика-эксперименты; контрольная группа 50/50.
- Светофор: 🟢 CR > 1.5% при p<0.05; 🟡 CR 1.0–1.5%; 🔴 < 1.0%.
- ICE: I=5, C=4, E=2 → **40**.

#### H-003 (problem)

> Mailing-трафик (1 377 визитов, **0 заявок**) не конвертит, потому что письма приходят на «уже знакомых» (Internal=3.0% CR) и не дают новый аргумент.

- Допущения: (1) база рассылки = старые лиды; (2) текст письма — общий, без сегментации; (3) визиты есть, значит ссылку нажимают.
- Методы: (a) проверить open/click rate в рассылочной платформе; (b) опрос подписчиков «почему не купили».
- Светофор: 🟢 если 50%+ из опрошенных скажут «не было нового повода».
- Дедлайн: 5 дней.
- ICE: I=3, C=5, E=4 → **60**.

#### H-004 (problem)

> Канал «Recommendation system» (100 визитов, 0 заявок) **поломан** — `ymRecommendation` подсасывает не релевантный трафик (Польша 42 визита, 0 конверсий).

- Допущения: (1) гео-таргетинг сломан; (2) рекомендации Метрики плохо работают для нишевой конференции; (3) можно выключить, не теряя оплат.
- Методы: (a) отключить на 7 дней и сравнить total visits; (b) посмотреть в Метрике, какие сайты-источники.
- Светофор: 🟢 total visits падает < 100/неделю, оплаты не меняются → выключаем.
- ICE: I=2, C=5, E=5 → **50**.

#### H-005 (solution)

> Команда ProductCamp **готова запустить B2B-аутрич по 30 целевым компаниям** (тех-конференции — Avito, Yandex, Tinkoff, VK, Wildberries), если конверсия из lead в paid > 10% (хотя бы 3 сделки за 14 дней).

- Допущения: (1) ICP — продакты-руководители трека; (2) у них есть бюджет на обучение; (3) персональное предложение конвертит лучше публичной цены.
- Методы: (a) LinkedIn + cold email + кастдев-звонок; (b) фиксировать в `b2b_manual`.
- Светофор: 🟢 ≥3 closed won за 14 дней; 🟡 1–2; 🔴 0.
- ICE: I=5, C=2, E=5 → **50**.

> Дополнительно: использовать `.claude/skills/synthetic-custdev/SKILL.md` для симуляции ответов целевой аудитории по каждой гипотезе перед реальным кастдевом.

### 8.1. Доверие к данным (must-have для v2.4.0)

- **Плашка происхождения данных** в шапке каждой страницы:
  `«Данные на {syncedAt}, счётчик {COUNTER_ID}, период {from}–{to}. Обновить →»`.

  `🟡 ДЕМО-РЕЖИМ — данные синтетические (pnpm seed), реальные значения см. после настройки счётчика`.

### 8.4. Доступность (WCAG AA, важно для отчётов в публику)

- Прогнать `axe-core` или Lighthouse a11y — на всех 9 страницах ≥ 90/100.
- Цветовые бейджи `🟢/🟡/🔴` дублируются текстом (сейчас есть, но проверить контраст в DOCX/PDF).

### 8.5. Производительность

- На /overview сейчас 7+ ECharts инстансов одновременно; ленивая загрузка по viewport (IntersectionObserver).
- Снапшот 11 000+ строк JSON — на /history превью загружается несколько секунд. Хранить «summary» в отдельной колонке `snapshot_summary` для быстрого list-view, полный `snapshot` тянуть только при клике.
- Server-side pagination для всех таблиц >100 строк (UTM, страницы, history).
- SQLite WAL-mode (`PRAGMA journal_mode=WAL`) для конкурентного чтения дашборда и sync.
- Бенчмарк: дашборд должен открываться за <2с при 10 000 строк channel_stats.

### 8.6. Безопасность / приватность

### 8.8. Тестовая стратегия (закрыть пробелы)

- **Property-based tests** для всех формул (CR ∈ [0,1], gap = max(0, goal − paid), etc.) через `fast-check`.

- **Snapshot stability tests:** генерим снапшот 100 раз на одних данных — все 100 байт-идентичны.
- **Mutation testing** через `stryker-mutator` хотя бы для `@pca/shared` (валидатор гипотез, ICE).
- **Migration tests:** каждая миграция применяется + откатывается + повторно применяется без ошибок.
