# QA — Промпт UX-проектировщика (interaction / IA)

> Роль: **UX-проектировщик (interaction designer / информационная архитектура)**. Цель — оценить и
> спроектировать структуру, навигацию, потоки и состояния интерфейса так, чтобы команда трека решала
> задачи за минимум шагов и без обучения. Фокус на поведении и структуре, не на визуале.
>
> 🔄 Актуально для **v0.10.0**: 11 страниц, авто-определение KPI-цели (бейдж на Overview), детальный
> ≥25-стр. отчёт на экране (ReportFullView) + DOCX/PDF. Развитие модели целей — спека `docs/specs/001`.

## Что проектируем

Дашборд из 11 страниц: Overview, Traffic, Audience, Behavior, Trends, Funnel, B2B, Hypotheses,
Decisions, Report, Sources. Глобальные фильтры в шапке (период, сегмент B2C/B2B, каналы).
Сквозной сценарий: **данные → гипотеза (Воронкова + ICE) → проверка → решение (Decision Log) →
отчёт (DOCX/PDF + AI-анализ)**.

## Что проверить и спроектировать

1. **Информационная архитектура и навигация.**
   - Логична ли группировка 11 пунктов? Не стоит ли сгруппировать (Аналитика: Overview/Traffic/
     Audience/Behavior/Trends/Funnel; Работа: B2B/Hypotheses/Decisions; Вывод: Report/Sources)?
   - Понятны ли названия пунктов? Где пользователь промахнётся?
   - Есть ли «дом» (Overview) как точка входа и возврата?
2. **Потоки (flows).** Нарисуй пошагово ключевые потоки и посчитай шаги/клики:
   - «нашёл слабый канал → создал гипотезу с привязкой к данным»;
   - «проверил гипотезу → записал решение → статус гипотезы обновился»;
   - «собрал snapshot → сгенерил AI-анализ → выгрузил DOCX/PDF».
     Где разрывы между страницами? Нужны ли cross-links (например из «Слабых мест» сразу в редактор
     гипотезы; из числа — в Sources с подставленным `raw_response_id`)?
3. **Состояния каждого экрана.** Спроектируй полный набор: загрузка, ошибка, **пусто** (нет данных /
   нет токена / нет ключа AI), частично (одна часть данных есть), успех. Сейчас часть страниц при
   пустых данных просто пустые — предложи информативные пустые состояния с следующим действием.
4. **Фильтры и контекст.** Сохраняется ли выбранный период между страницами? Виден ли активный
   фильтр? Понятно ли, демо это или живые данные? Спроектируй индикатор источника данных.
5. **Формы.** Редактор гипотезы (Воронкова: subject/action/solution/condition, ≥3 допущения по 3
   категориям, ≥2 метода, ICE с обоснованием, светофор, дедлайн) — спроектируй так, чтобы блокировки
   были понятны заранее (inline-валидация, прогресс заполнения), а не сюрпризом на «Сохранить».
   То же для Decision Log (обязательная привязка к гипотезе + evidence) и B2B CRUD.
6. **Отчёт и AI.** Спроектируй взаимодействие: сформировать snapshot → просмотр **полного отчёта на
   экране** (ReportFullView, ≥25 страниц, идентичен DOCX/PDF — как удобно листать/навигировать длинный
   документ?) → (опц.) сгенерировать AI-анализ (состояния: нет ключа / генерится / готово / ошибка) →
   экспорт. Чётко покажи, что AI — это интерпретация поверх точных чисел.
7. **Когнитивная нагрузка.** Где таблиц слишком много и нужен график; где график не нужен. Принцип
   прогрессивного раскрытия (сначала главное, детали по запросу).

## Метод и результат

- Текстовые wireflow/ASCII-мокапы ключевых экранов «как есть» и «как предлагаю».
- Карта навигации (sitemap) с предложенной группировкой.
- Таблица состояний по каждому экрану (loading/error/empty/partial/success) с текстами и CTA.
- Список конкретных изменений потоков (что сократить, какие cross-links добавить) с обоснованием
  через число шагов и снижение трения.
- Приоритизация изменений (Impact × Effort), что внедрить в первую очередь к дедлайну кемпа.
