## 1. Резюме (½ страницы)

Сегодня структура — **плоский список из 9 пунктов** без визуальной группировки. Это создаёт
ощущение «9 равнозначных страниц», тогда как реальные задачи делятся на три кластера:
анализ → действие → управление. Главное «провисание» структуры:

- **Нет страниц «действия» (Hypotheses / Decisions / B2B)** — пользователь, посмотрев данные,
  возвращается в Excel и теряется. Это IA-провал, а не только feature gap.
- **Контекст фильтра «прыгает»** — на десктопе он есть на 7 страницах из 9 (логично), но при
  переходе с /traffic → /report пользователь не уверен, относятся ли числа отчёта к тому же
  периоду.
- **Пустые состояния отсутствуют** на /history, /report, /hypotheses-API (нет UI), /b2b — везде
  «голый» экран без подсказок next-action.
- **Формы не реализованы** — Settings есть, но это форма для админа. Главной формы (гипотеза по
  Воронковой) — нет, а это **центральная JTBD продукта**.

## 2. Карта навигации — «как есть» vs «предлагаю»

### Как есть (flat 9)

```
ProductCamp · Конверсии  [Обзор] [Трафик] [Поведение] [Воронка] [Цели] [Отчёт] [История] [Настройки] [Справка]
```

### Предлагаю (3 кластера, видимая группировка)

```
ProductCamp · Конверсии · v2.3.0
┌──────────────────────────────────────────────────────────────────────────────┐
│ АНАЛИЗ:  Обзор · Трафик · Поведение · Воронка · Цели                          │
│ ДЕЙСТВИЕ:  Гипотезы · Решения (DL) · B2B-пайплайн                             │
│ ВЫВОД:  Отчёт · История                                                      │
│                                          ⚙ Настройки · ❓ Справка             │
└──────────────────────────────────────────────────────────────────────────────┘
```

Группы — это **визуальные секции в шапке** (тонкие разделители или подписи мелким шрифтом
над пунктами). На мобилке — в гамбургере те же три кластера + collapse-секции.

Преимущества:

- Новичок-волонтёр сразу видит, что есть «3 этапа» работы.
- Снижается риск «куда мне теперь — на /report или на /history?»
- Структура совпадает с циклом Double Diamond из `docs/methodology-double-diamond.md`.

### Sitemap v2

```
/                              → Overview (KPI + смысл цели)
/analysis/traffic              ← /traffic
/analysis/behavior             ← /behavior
/analysis/funnel               ← /funnel
/analysis/goals                ← /goals
/act/hypotheses                NEW
/act/hypotheses/:id            NEW (детальная карточка + история проверок)
/act/decisions                 NEW (Decision Log)
/act/b2b                       NEW (CRUD сделок)
/output/report                 ← /report
/output/report/:snapshotId     ← /report?snapshot=...
/output/history                ← /history
/settings                      без изменений
/help                          + раздел «10 минут без брифа»
/admin/adoption                NEW (внутр., usage_events)
```

Если миграция URL-ов слишком инвазивна — оставить плоские `/hypotheses`, `/decisions`, `/b2b`,
а группировка только в навигации.

## 3. Сквозные потоки (wireflow)

### Поток 1 — «нашёл слабый канал → создал гипотезу»

**Как сейчас:**

```
/overview [клик «Слабые места»] → видит «Direct CR 0.6%»
   → ??? (некуда идти, гипотезу создать нельзя)
```

3+ клика, теряется.

**Как предлагаю:**

```
/overview блок «Слабые места»
   → каждая строка кликабельна
   → переход на /act/hypotheses/new?source=overview&channel=Direct&metric=cr&value=0.6
   → форма Воронковой предзаполнена:
       subject: "Целевая аудитория Direct"
       action: "готова сделать оплату"
       solution: "(вписать)"
       condition: "(вписать)"
       assumption[0]: "CR 0.6% при 37 649 визитах — потенциал в +N оплат" (cite snapshot)
   → snapshot привязан, evidence уже подгружен
```

2 клика. Главное — **глубокая ссылка с предзаполнением**.

### Поток 2 — «проверил гипотезу → решение → отчёт»

**Как предлагаю:**

```
/act/hypotheses/H-007 → кнопка «Сменить статус» → drawer:
   ├─ статус (🟢/🟡/🔴)
   ├─ evidence: какой snapshot, какие raw_response_id
   ├─ автоген запись DL → ссылка на /act/decisions/DL-NNN
/act/decisions/DL-NNN → автопредложение «Добавить в недельный отчёт?»
   → /output/report (формирование с пометкой «упомянуть DL-NNN»)
```

### Поток 3 — «снапшот → AI → экспорт»

**Как сейчас:**

```
/report → «Сформировать срез данных» → видишь снапшот →
  → отдельная кнопка «Сгенерировать AI» (~30-60с без прогресса)
  → отдельные кнопки «Export DOCX» / «Export PDF» (всегда скачивают, нельзя превью)
```

**Как предлагаю:**

```
/report → кнопка «Сформировать срез»
   → ① строится снапшот (1-3с прогресс)
   → ② автоматически предлагает: «Хотите AI-анализ? (~45с, тратит {tokens})»
   → ③ если да — прогресс-бар 5 секций (Контекст → Каналы → UTM → Воронка → Рекомендации)
   → ④ превью отчёта прокручивается **с TOC слева** (плавающее оглавление)
   → ⑤ экспорт: dropdown [📄 DOCX] [📕 PDF] [🔗 Скопировать ссылку] [📧 Отправить по email]
```

### Поток 4 — «выбор периода»

**Контекст-проблема:** пользователь нажимает 30д → видит цифры → переходит на /report
→ непонятно, отчёт будет по тем же 30 дням?

**Как предлагаю:** в шапке /report (и /history) **persistent badge** «Период: 14 мая — 27 мая
(14 дней) · применяется ко всем расчётам». При наведении — tooltip «Изменить → шапка». При
скролле — badge становится sticky.

### Поток 5 — «волонтёр без брифа за 10 минут»

**Как предлагаю:** на /overview при первом визите (localStorage флаг `firstVisit`) — **dismissable
onboarding-карточка** наверху:

```
👋 Привет! Это аналитический дашборд ProductCamp.
   3 шага, чтобы начать:
   ① Посмотри KPI на этой странице
   ② Найди «слабое место» в блоке ниже
   ③ Создай гипотезу → действуй

   [Показать тур (45с) →]    [Спрятать]
```

## 5. Формы — главное проектное решение

### Форма «Новая гипотеза» (Воронкова + ICE)

Layout — **stepper из 5 шагов** (а не один длинный экран):

```
Шаг 1/5 · Сформулировать
   ┌──────────────────────────────────────────────────────────────┐
   │ Шаблон: {ЦА} {готова Х} {решение}, если {условие}             │
   │  ЦА:        [_______________________]                         │
   │  Готова:    [сделать ▼] [заплатить за] [переключиться на]     │
   │  Решение:   [_______________________]                         │
   │  Если:      [_______________________]                         │
   │                                            [Далее →]          │
   └──────────────────────────────────────────────────────────────┘
   ✅ Inline-валидация: показывает собранную фразу под полем

Шаг 2/5 · Допущения (≥3 в 3 категориях)
   ┌───────────────┬────────────────────────────────────────────┐
   │ 🔴 Риск       │ [_______________________________________]  │
   │ 🟡 Гипотеза   │ [_______________________________________]  │
   │ 🟢 Неизвестное│ [_______________________________________]  │
   └───────────────┴────────────────────────────────────────────┘
   + [Добавить допущение]
   Прогресс: 3/3 минимум · можно больше

Шаг 3/5 · Методы проверки (≥2)
   ┌──────────────────────────────────────────────────────────────┐
   │ Метод 1: [Кастдев ▼]   Метрика: [____]   Срок: [____]         │
   │ Метод 2: [A/B-тест ▼]  Метрика: [____]   Срок: [____]         │
   └──────────────────────────────────────────────────────────────┘

Шаг 4/5 · ICE
   ┌──────────────────────────────────────────────────────────────┐
   │ Impact:     [▮▮▮▮▯] 4   Почему: [______________________]      │
   │ Confidence: [▮▮▮▯▯] 3   Почему: [______________________]      │
   │ Ease:       [▮▮▮▮▯] 4   Почему: [______________________]      │
   │                                                                │
   │ ИТОГ: I × C × E = 4 × 3 × 4 = 48                              │
   │ (важно: произведение, не среднее — см. ADR-005)               │
   └──────────────────────────────────────────────────────────────┘

Шаг 5/5 · Светофор и дедлайн
   ┌──────────────────────────────────────────────────────────────┐
   │ 🟢 Зелёный свет, если:  [______________________]              │
   │ 🟡 Жёлтый, если:        [______________________]              │
   │ 🔴 Красный, если:       [______________________]              │
   │ Дедлайн решения:        [📅 ____-__-__]                        │
   │                                                                │
   │ Привязать snapshot: [autodetect: 81f80a4b… ▼] [сменить]       │
   │                                                                │
   │                        [← Назад]  [💾 Сохранить как DRAFT]    │
   └──────────────────────────────────────────────────────────────┘
```

**Ключевые UX-принципы:**

- Прогресс-бар сверху всегда виден.
- Inline-ошибки появляются **сразу** под полем, не на «Сохранить».
- Кнопка «Сохранить как DRAFT» на любом шаге — не теряется работа.
- На каждом шаге — мини-подсказка из `.claude/skills/hypothesis-check/SKILL.md`.

### Форма Settings

Реструктурировать в **3 группы (collapse-секции)**:

1. **Источник данных** (открыто): COUNTER_ID, GOAL_ID, OAuth — то, с чего начинают.
2. **AI и нарративы** (collapsed по умолчанию): ANTHROPIC_API_KEY.
3. **Расширенное** (collapsed): Client ID/Secret, очистка кеша, реиндексация.

Сверху — **Sticky-сводка** «Текущий счётчик <COUNTER_ID> · Цель: Ecommerce: покупка · AI: подключен ✅».

### Форма GOAL_ID select

Сейчас select из всех целей (включая архивные) — длинный. Улучшение:

- **Группировка**: «Активные» / «Архивные».
- **Bage у каждой цели**: тип (`e_purchase` = «Покупка», `messenger` = «Мессенджер», и т.д.).
- **Search-input** сверху.
- Под select — preview «Если выбрать эту цель, на дашборде увидите: ~417 заявок за 14 дней».

## 6. Когнитивная нагрузка — где режем, где добавляем

### Резать (информационный шум)

- **Топ страниц входа на /overview** — 5 строк, в 4 из которых одинаковый URL → агрегировать.
- **7 бейджей «значительно ниже среднего» подряд** — оставить топ-3 «требующих внимания»,
  остальные свернуть в «показать ещё 4».
- **Снапшот «Каналов: 2293»** — заменить на «Каналов: 8, дней: 14, точек данных: 112».
- **Длинная UTM-таблица на /overview** — оставить топ-5, ссылка «Полная таблица → /traffic».
- **History — 20+ снапшотов скроллом** — пагинация по 10 + фильтр по периоду.

### Добавить (где не хватает)

- **WoW дельта** на KPI: «417 заявок (+23 vs прошлая неделя)».
- **Sparkline в каждой строке UTM-таблицы** — мини-график тренда визитов.
- **Кнопка «Объяснить» рядом с цифрой** — открывает drawer с формулой и raw_response_id.
- **«Связанные действия»** в конце каждой страницы: «На основе этих данных можно создать
  гипотезу о \_\_\_ →» (CTA).

## 7. Контекст фильтра — индикатор источника данных

Сейчас непонятно, где данные «живые», где «демо», где «закешированные». Предлагаю
**глобальный header-badge** под навигацией:

```
🟢 Live · Метрика <COUNTER_ID> · synced 27 мая 03:37 · период 14–27 мая · сегмент B2C+B2B  [Refresh]
```

Цвета бейджа:

- 🟢 Live (последний sync < 1 час).
- 🟡 Stale (1–24 часа) — «Обновить?» CTA.
- 🔴 Outdated (> 24ч) — «Данные старые, обновите перед презентацией».
- ⚪ Demo (COUNTER_ID=0) — «Синтетические данные, см. /settings».

## 9. Что НЕ проектировали в этом прогоне

- Визуальный дизайн (цвета, типографика, иконки) — отдельный отчёт визуального дизайнера.
- Полные мокапы каждого экрана — этот документ wireflow-уровня.
- Mobile-pass пиксельно — нужен живой devtools mobile прогон (см. m-021 в qa-report).
- A/B-тестирование самих UX-решений — после внедрения.
