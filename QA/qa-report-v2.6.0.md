### 🟠 C-001 [Critical] — /hypotheses реализована не по спеке методологии Воронковой

**Где:** http://localhost:5173/hypotheses
**Симптом:** На странице **только одна кнопка** «Сгенерировать гипотезы» (AI-генератор). Отсутствует:

- Stepper-форма из 5 шагов (Сформулировать → Допущения → Методы → ICE → Светофор+дедлайн+snapshot).
- Список существующих гипотез (`/api/hypotheses` возвращает `[]`).
- Валидация формата фразы «{ЦА} {готова Х} {решение}, если {условие}».
- Поля ≥3 допущений (риск/гипотеза/неизвестное), ≥2 метода, светофор, дедлайн, ICE = I×C×E.
- Привязка к snapshot.
- Кнопка «Сохранить как DRAFT».

**Нарушает:** `CLAUDE.md` § «Гипотеза без формата гипотезы — не гипотеза», `fix-prompt-v2.5.0` §2.2, `docs/specs/011-hypothesis-form-stepper.md` (если спека написана).

**Влияние:** Ключевая JTBD ЦА — «структурированно создать гипотезу» — не закрывается. Цепная реакция:

- На /decisions dropdown «Гипотеза» пустой → создать решение невозможно.
- Decision Log остаётся пустым.
- Цикл «данные → гипотеза → проверка → решение → отчёт» обрывается на /hypotheses.
- Зелёный свет из `CLAUDE.md` («≥5+5 гипотез, ≥3 решения») недостижим.

**Должно быть:** Stepper-форма по `fix-prompt-v2.4.0.md §2.2` (react-hook-form + zod, draft на каждом шаге, подсказки из `.claude/skills/hypothesis-check/SKILL.md`). AI-генератор — опциональная фича, не замена.

**Воспроизведение:**

### 🟡 M-002 [Major] — Микро-расхождение визитов между страницами

**Где:**

- /goals: «Визитов 4 912»
- /funnel: «Визитов 4 912»
- /behavior: «Визитов 4 984»

**Симптом:** Behavior 4 984 ≠ Goals/Funnel 4 912 (разница 72 визита, ~1.5%). Не катастрофа как в v2.3.0 (4 925 vs 57 034), но **factsource всё ещё не единый**.

**Гипотеза причины:** Behavior считает визиты как `SUM(page_stats.visits)` (или с дедупом по странице), а Goals/Funnel — как `SUM(channel_stats.visits)`. У Метрики оба источника технически разные (визиты по URL входа vs по каналам), но **пользователь ожидает одно число**.

**Должно быть:**

- Единая функция `getVisitsForPeriod(from, to, segment)` в backend, использовать ВЕЗДЕ.
- Inv-тест: `code/backend/tests/factsource.test.ts` — `expect(Behavior.visits).toBe(Funnel.visits).toBe(Goals.visits)`.

**Нарушает:** инвариант 0 регресса «согласованность чисел между страницами».

### 🟡 M-004 [Major] — UTM-покрытие 0.0% подозрительно низкое

**Где:** /traffic и /overview — «Низкое покрытие UTM: 0.0% (порог 70%) — часть трафика не атрибутирована».

**Симптом:** В Метрике видна UTM-разметка (на /overview UTM-таблица заполнена кампаниями tg/infopartner/codedaria и т.д.), а покрытие 0% означает «ни один визит не имеет UTM». Противоречие.

**Гипотеза причины:** Формула `utmCoverage(channels) = utmVisits / totalVisits` где `utmVisits` берётся не оттуда — возможно, считается только из `channel_stats.utmCoverage`, а не из `utm_stats`.

**Должно быть:**

- Проверить формулу в `code/backend/src/analytics/utm-coverage.ts` (или где она).
- Если в `utm_stats` есть строки — `utmCoverage` не может быть строго 0%.
- Tooltip ⓘ «как считается»: `(SUM(utm_stats.visits) / SUM(channel_stats.visits)) * 100`.
