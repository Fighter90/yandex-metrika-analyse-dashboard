# UX-анализ ProductCamp Conversion Analytics Tool v2.3.0

> Роль: UX-аналитик. Цель — оценить, насколько продукт реально решает задачи команды трека
> «Конверсии и лидген» ProductCamp, найти разрывы между поведением/потребностями ЦА и
> текущим UX. Прогон выполнен на живых данных счётчика <COUNTER_ID>, период 2026-05-14 — 2026-05-27.

---

## 1. Резюме (½ страницы) — решает ли продукт главную задачу?

**Главный пользовательский вопрос: «Откуда приходят платные билеты и как дойти до 300?»**

**Вердикт: 🟡 ЧАСТИЧНО.** Продукт **показывает**, откуда приходит трафик и заявки (Overview, Traffic,
UTM-таблица — это рабочий уровень). Но провисает в трёх местах, без которых задача
«дойти до 300» не решается:

1. **Доверие к данным сломано:** одна и та же метрика «визиты за период» имеет
   на разных страницах разные значения (Behavior 4 925 vs Funnel/Goals 57 034), а в снапшоте
   тысячи дублей строк → пользователь сразу теряет веру в инструмент.
2. **«Заявка ≠ оплата» декларируется, но логика смешана:** primary_goal = `Ecommerce: покупка`
   (то есть это уже оплата), но UI продолжает называть её «заявкой» и оценивать через зашитый
   коэффициент 0.3 → числа без объяснения, источник эвристики не виден.
3. **Ядро методологии — гипотезы и Decision Log — недоступны через UI:** API есть,
   страниц нет. Цикл «данные → гипотеза → решение» обрывается на дашборде.

Без п.1–3 продукт остаётся «дашбордом просмотра», а не «инструментом, который доводит до 300».

## 3. Топ-10 находок по серьёзности (NN-шкала 0–4)

> Серьёзность: 4 = катастрофа (пользователь не выполнит задачу), 3 = высокая,
> 2 = средняя, 1 = косметика, 0 = подсказка.

### F-1 [4] Числа не сходятся между страницами → подрыв доверия

- **Где:** Behavior «4 925 визитов» vs Funnel «57 034 визитов» за один период.
- **Влияет на:** все задачи доверия к данным.
- **Гипотеза улучшения:** единый «factsource» в backend + плашка «трассировка к raw_responses» на каждой карточке (см. `qa/fix-prompt-v2.3.0.md` §1.2).
- **Метрика:** % пользователей, кто прошёл по 3+ страницам подряд и не закрыл вкладку.

### F-2 [4] Нет страниц Hypotheses / Decisions / B2B

- **Где:** меню (/ overview / traffic / behavior / funnel / goals / report / history / settings / help) — всё.
- **Влияет на:** ключевая ценность продукта по `CLAUDE.md` («гипотеза без формата гипотезы — не гипотеза», Decision Log замыкает цикл).
- **Гипотеза:** добавить страницы с формой-валидатором Воронковой и kanban-доской статусов.
- **Метрика:** ≥5 problem-гипотез и ≥5 solution-гипотез за первую неделю; ≥3 записи в DL.

### F-3 [4] CR > 100% (200% на /reg-new#popup:myform)

- **Где:** /behavior блок «Ключевые наблюдения», бейдж «✅ Лучший CR».
- **Влияет на:** пользователь видит «✅ Лучший» с заведомо невозможным числом → теряет понимание метрики.
- **Гипотеза:** инвариант `cr ∈ [0, 1]` + если больше — переименовать в «events/visit».
- **Метрика:** 0 случаев CR > 100% на проде.

### F-4 [3] Дубли строк в отчёте → нечитаемо

- **Где:** /report?snapshot=... блок «Полный отчёт», 13+ идентичных строк подряд (Direct/Internal/Link/Mailing).
- **Влияет на:** задача «собрать отчёт для стейкхолдеров» — текущий вывод нельзя показать.
- **Гипотеза:** дедуп на уровне ETL (UNIQUE INDEX) + агрегация по канал×день.
- **Метрика:** в снапшоте за 14 дней ≤ (8 каналов × 14 дней) = 112 строк channel_stats.

### F-5 [3] Терминология «Заявок B2C» при цели `e_purchase`

- **Где:** Overview KPI, Goals «B2C заявки ≈ 125 платных», бейдж «заявка ≠ оплата».
- **Влияет на:** доверие; пользователь не понимает, какое число настоящее.
- **Гипотеза:** `formatGoalLabel(goal)` — если `e_purchase` → «Оплат» и убрать оценку ×0.3.
- **Метрика:** survey «понимаете ли вы, что показывает KPI?» — ≥4/5.

### F-6 [3] UTM-покрытие противоречит между страницами

- **Где:** Overview 100% «✅ хорошая атрибуция» vs Traffic 0.0% «⚠ ниже порога 70%».
- **Влияет на:** доверие; пользователь не знает, какому числу верить.

### F-7 [3] Bar-chart «Каналы — визиты» на /traffic — пустой/обрезанный

- **Где:** /traffic первый график.
- **Влияет на:** первое впечатление от страницы Трафик.
- **Гипотеза:** убрать top-N обрезание, добавить tooltip.

### F-8 [2] Эвристика «×30%» в коде без объяснения

- **Где:** Goals «≈ 125 платных (оценка 30% конверсии заявка→оплата)».
- **Влияет на:** интерпретация; нет ссылки на ADR, нельзя поменять без правки кода.
- **Гипотеза:** настройка в Settings + ADR.

### F-9 [2] История отчётов — UUID без контекста

- **Где:** /history.
- **Влияет на:** возврат к работе.
- **Гипотеза:** добавить колонки «AI-анализ есть/нет», «связано с гипотезой», поиск, удаление.

### F-10 [2] Mailing/Messenger всегда наказаны жёстким порогом 2%

- **Где:** /overview бейджи.
- **Влияет на:** действие — пользователь видит 7 красных бейджей и не знает, на каком сосредоточиться.
- **Гипотеза:** per-channel benchmark.

## 4. Какие события/метрики добавить, чтобы измерять принятие и пользу

Сейчас в продукте **нет product analytics на сам инструмент**. Без них нельзя измерить
зелёный/жёлтый/красный свет из `CLAUDE.md`. Минимум:

| Событие                                                        | Когда                            | Зачем                          |
| -------------------------------------------------------------- | -------------------------------- | ------------------------------ |
| `pca.session.start`                                            | загрузка любой страницы дашборда | DAU/WAU команды                |
| `pca.page.view` `{page}`                                       | смена роута                      | какие страницы реально нужны   |
| `pca.filter.change` `{preset, segment, archived}`              | изменение фильтра                | пользователи фильтруют или нет |
| `pca.sync.start/finish` `{durationMs, stages}`                 | `POST /api/settings/refresh`     | реальные времена sync          |
| `pca.hypothesis.create` `{type: problem/solution, iceScore}`   | сохранение гипотезы              | ≥5+5 для зелёного света        |
| `pca.decision.create` `{linkedHypothesisId, snapshotId}`       | сохранение DL                    | ≥3 для зелёного                |
| `pca.report.generate` `{format: docx/pdf, snapshotId, withAi}` | экспорт отчёта                   | какой формат востребован       |
| `pca.ai.insights.generate` `{tokens, durationMs, success}`     | `POST /api/report/insights`      | стоимость и польза AI          |
| `pca.search.empty` `{query}`                                   | пустой результат                 | дыры в наполнении              |
| `pca.error.shown` `{message, page}`                            | алерт пользователю               | где нужны улучшения            |

Хранить в той же SQLite в таблице `usage_events(id, name, payload_json, user_id, created_at)`,
выгружать в дашборд «Принятие инструмента» (внутренний, отдельный роут /admin/adoption).

## 6. Онбординг волонтёра — пройдёт ли «10 минут без брифа»?

**Ответ: НЕТ, не пройдёт.** Что мешает:

1. **Главная задача волонтёра — внести гипотезу** — недоступна (нет страницы).
2. На Overview новичок видит 7 красных бейджей «значительно ниже среднего» подряд — паника, непонятно, на каком сфокусироваться.
3. KPI-метрики называются «Заявок B2C», но дальше сказано «заявка ≠ оплата» — волонтёр не понимает, на что смотрит.
4. /history показывает 20+ UUID без подписи периода в человеческом виде — невозможно понять, какой снапшот «вчерашний».
5. /settings первое, что показывает, — длинный список OAuth-полей, COUNTER_ID, GOAL_ID select. Волонтёр пугается «я не должен ничего здесь трогать».

**Что добавить:**

- На /help — короткий чек-лист «Сценарий волонтёра за 10 минут» с шагами и якорными ссылками на страницы.
- На Overview — баннер сверху «Здесь главные KPI. Если видите проблему — перейдите на /traffic → /behavior → /hypotheses» (когда страница появится).
- Пустые состояния на /history, /report: «У вас 0 гипотез — начните с …» (CTA на /hypotheses).
- На /settings вверху плашка «🔒 Эти настройки трогать не нужно, если их уже настроил админ».

## 7. Мобильная версия

Проверил кнопку гамбургер-меню — она присутствует в DOM (`data-mobile-menu`),
но live mobile прогон в этом сеансе не довели (Chrome MCP не сжимает viewport до 380px).

**На что обратить внимание при mobile-pass:**

- KPI-карточки на /overview — 4 в строку на десктопе. На мобилке должны переноситься в 2×2 или 1×4.
- Таблицы UTM/страниц входа — длинные горизонтально. Нужен горизонтальный скролл или адаптивная вёрстка (карточки на мобиле).
- ECharts графики ресайзятся ли при resize окна? Проверить `chart.resize()` на window.resize.
- FilterBar — пресеты 7д/14д/30д/90д/1г + сегмент + кастомный picker + чекбокс архивных — 6+ элементов. На мобилке скорее всего перепрыгивают строки.
- Settings — длинная вертикальная форма; OK, но прогресс-бар sync должен быть липким (sticky), чтобы видеть статус при скролле.

## 8. Что я НЕ проверил (для следующего прогона)

- Кастдев с тремя реальными пользователями (Лиза + 2 волонтёра) — времянный лимит регрессии.
- Sync с прогресс-баром live (не запускал, чтобы не сбить данные).
- Полноценный мобильный прогон в DevTools (узкий viewport).
- A/B сравнение DOCX vs PDF на одном snapshotId (sandbox не видит localhost для скачивания).

## 9. Главный совет для команды

**Не катите v2.4.0 без страниц Hypotheses + Decisions + B2B-CRUD.** Без них продукт остаётся
дашбордом. Параллельно — починить «доверие к данным» (F-1, F-3, F-4, F-5, F-6), потому что
красивая навигация по дырявым числам не имеет ценности.
