# QA — Промпт UX/визуального дизайнера

> Роль: **UX-дизайнер (visual / UI designer)**. Цель — оценить и улучшить визуальное исполнение
> дашборда: типографику, цвет, сетку, графики и доступность так, чтобы данные читались мгновенно и
> «наглядно», а интерфейс выглядел профессионально и доверительно. Стек: React 18 + Tailwind +
> Apache ECharts.
>
> 🔄 Актуально для **v2.3.0**: 9 страниц дашборда, мобильное меню (гамбургер), фильтры до 1 года
> (7д/14д/30д/90д/1г), GOAL_ID select, AI-анализ 5 секций с HTML-рендерингом (таблицы, заголовки,
> форматирование), прогресс-бар синхронизации (10 стадий), History → «Просмотреть», FilterBar
> скрыт на /help и /settings, цветные бейджи (🟢/🟡/🔴) с рекомендациями на каждой странице.

## Что оценить

1. **Визуальная иерархия.** На каждой из 9 страниц: ведёт ли глаз к главному (KPI, «слабые места»,
   воронка)? Достаточен ли контраст размеров/веса заголовков, KPI-чисел и второстепенного текста?
2. **Сетка и ритм.** Единые отступы, выравнивание карточек, ширина контента, поведение на узких
   экранах (графики/таблицы не должны ломаться). Проверь grid из карточек. Мобильная версия:
   не ломаются ли графики при < 768px?
3. **Графики (ECharts) — наглядность.**
   - Подписи осей, легенды, единицы (%, доли, визиты), тултипы, форматирование чисел (разделители).
   - Согласованная палитра: один смысл — один цвет (визиты vs заявки везде одинаково).
   - Воронка, линия трендов, бары топ-каналов/стран/страниц, донат устройств, микс каналов —
     читаются ли «с одного взгляда»? Где график перегружен или, наоборот, не хватает визуала.
   - Сокращение длинных URL/названий без потери смысла; пустые/одно-точечные данные выглядят ок.
4. **Цвет и семантика.** Светофор гипотез 🟢🟡🔴, бейджи рекомендаций, бейджи статусов,
   бейдж низкого UTM-покрытия — единая, доступная семантика. «Заявка ≠ оплата» должно читаться
   визуально (разные акценты для заявок и оплат).
5. **Типографика.** Шрифтовая шкала, межстрочные интервалы, читаемость таблиц (выравнивание чисел по
   правому краю, моноширинность для сумм), длина строки текста (AI-анализ в HTML — хорошо ли читается?).
6. **Доступность (a11y).** Контраст по WCAG AA, не только цветом передавать смысл (иконки/подписи к
   светофору и дельтам), фокус-стейты, размеры кликабельных зон, aria для интерактивных чартов,
   читаемость при 200% зуме. Мобильный гамбургер: доступны ли все пункты навигации?
7. **Состояния и обратная связь.** Визуал загрузки/ошибки/пустого состояния, состояния кнопок
   (Export DOCX/PDF, «Сгенерировать AI-анализ», «Перестроить отчёт»: idle/disabled/loading/done).
   Прогресс-бар AI-анализа (5 стадий) — понятен ли? Прогресс-бар синхронизации (10 стадий) — понятен ли?
8. **Доверие и бренд.** Аккуратность, консистентность компонентов Tailwind, единый стиль карточек,
   подача отчёта (DOCX/PDF) как «презентабельного» артефакта для стейкхолдеров.
9. **Страница «Справка».** Визуальная подача документации: легко ли читать длинные разделы? Хватает ли
   визуального разделения между секциями? Иконки (📊🔗📄🔻🎯📋📅⚙️❓📖) — помогают навигации?

## Метод и результат

- Пройди все страницы на демо- и живых данных; делай скриншоты «как есть».
- Для каждой находки: экран, проблема, влияние на читаемость/доверие/доступность, **серьёзность**
  (0–4), предложение (с конкретикой: размеры, цвета-токены, отступы, опции ECharts).
- Предложи мини-дизайн-систему: палитра (с семантическими токенами visits/applications/paid/
  warning/success/danger), типографическая шкала, отступы, стиль карточек и графиков — чтобы всё было
  единообразно «наглядно».
- Чек-лист доступности (WCAG AA) с пройдено/провалено по каждому пункту.
- Приоритизированный список визуальных правок (Impact × Effort) и быстрые победы, которые повышают
  наглядность за минимальные усилия (легенды, единицы, форматирование чисел, единая палитра).
