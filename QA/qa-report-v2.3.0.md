## Дефекты (по приоритетам)

### 🔴 BLOCKER (нельзя катить)

#### B-001 — Падает `pnpm typecheck` (build broken)

- **Файл:** `code/backend/src/report/docx/builder.ts:72`
- **Симптом:** `TS2820: Type '"CLEAR"' is not assignable to type '"clear" | ...'. Did you mean '"clear"'?`
- **Фактически:** TypeScript-сборка backend падает, поэтому весь шаг build/release валит CI.
- **Должно быть:** `type: 'clear'` (lowercase).
- **Воспроизведение:** `pnpm typecheck` из корня репо.
- **Нарушенный инвариант:** quality-gate «build зелёный перед merge».

#### B-002 — Числа визитов не сходятся между страницами (anti-hallucination нарушен)

- **Страницы:** Overview chart ~5–6 тыс/день, Behavior «Визитов 4 925», Funnel «Визитов 57 034», Goals «Визитов 57 034», Снапшот «Суммарно визитов в выборке каналов: 57034».
- **Симптом:** одна и та же метрика «визиты за период» имеет минимум **2 разных значения** (4 925 vs 57 034) на разных страницах дашборда. Один и тот же snapshotId не должен порождать разные числа.
- **Должно быть:** единый источник правды (`raw_responses` → `channel_stats`), одно число на все страницы.
- **Нарушенный инвариант:** «любая цифра прослеживается до `raw_responses`», «один snapshotId → идентичный контент».

#### B-003 — CR > 100% на странице Behavior (математически невозможно)

- **Страница:** `/behavior`, блок «Ключевые наблюдения» и «Конверсия страниц входа».
- **Симптом:** `✅ Лучший CR: /reg-new#popup:myform 200.0%`. CR = заявки/визиты не может превышать 1.0 (100%), кроме случаев одна форма даёт несколько целевых событий — но тогда нужно использовать другой знаменатель.
- **Должно быть:** либо метрика называется не «CR» (например «events per visit»), либо знаменатель/числитель пересчитать.
- **Нарушенный инвариант:** anti-hallucination и здравый смысл.

### 🟠 CRITICAL

#### C-004 — Дубликаты строк в снапшоте/отчёте (тысячи копий одной строки)

- **Страница:** `/report?snapshot=...` блок «Полный отчёт».
- **Симптом:** за 2026-05-14 одна и та же строка «Direct traffic: визитов 196, заявок 1 (CR 0.5%), отказы 19.4%» повторена **13+ раз подряд**. То же для Ad/Internal/Link/Mailing/Messenger/Social. Снапшот сообщает «Каналов: 2293» — нереалистично для 7–10 реальных каналов за 14 дней.
- **Похоже:** агрегация `channel_stats` идёт **по запросу** (queryHash, дата), вместо `GROUP BY (date, channel)`, и дублирует строки на каждый sync.
- **Должно быть:** UNIQUE INDEX `(date, channel)` или агрегация при чтении.
- **Нарушенный инвариант:** anti-hallucination, детерминизм.

#### C-005 — Дубли страниц входа (одинаковый URL × N)

- **Страница:** `/` блок «Топ страниц входа»: 5 строк, в 4-х из них `https://productcamp.ru/` с разными визитами/bounce/заявками.
- **Симптом:** одинаковый URL фигурирует несколько раз — значит ETL не дедуплицирует, либо учитывает разные query/fragment как разные «страницы».
- **Должно быть:** ETL либо нормализует URL (без query/fragment), либо UI разделяет дубли тегом исходного `ym:s:startURL` vs `ym:s:URL`.

#### C-006 — Противоречие «UTM покрытие» между страницами

- **Overview:** зелёный бейдж «✅ UTM покрытие 100% — хорошая атрибуция».
- **Traffic:** жёлтый блок «Низкое покрытие UTM: 0.0% (порог 70%)».
- **Симптом:** одна и та же метрика, противоположные оценки.
- **Должно быть:** одно значение, или явно разные определения с пояснением.

#### C-007 — Bar chart «Каналы — визиты» на Traffic обрезан до 3 каналов из 7

- **Страница:** `/traffic`, первый график.
- **Симптом:** на оси X нарисованы только Direct/Internal/Messenger, столбцов не видно вовсе (масштаб шкалы 0–40 000, но столбцы нулевые), хотя ниже есть бейджи на 7 каналов с реальными визитами (Direct, Link, Mailing, Social, Messenger, Ad, Recommendation, Internal).
- **Должно быть:** все каналы с визитами > 0 отрисованы.

#### C-008 — Терминология primary_goal vs UI labels («Заявок B2C 417» при `e_purchase`)

- **API:** `/api/metrics/primary-goal` → `{type: 'e_purchase', name: 'Ecommerce: покупка'}` — то есть цель = **покупка**, goal_reaches = реальные оплаты.
- **UI:** на Overview подпись «Заявок B2C 417 · заявка ≠ оплата», на Goals «B2C заявки (Метрика): 417 ≈ 125 платных (оценка 30% конверсии заявка→оплата)».
- **Симптом:** если goal — это уже оплата, нет смысла дополнительно делить на 30%. И наоборот, если это заявки, primary_goal type не должен быть `e_purchase`.
- **Должно быть:** если goal.type ∈ {`e_purchase`} → метка «Оплат» и убрать оценку «×0.3»; если это форма/заявка → не показывать как покупку.

#### C-009 — Гипотезы/Decisions/B2B пустые: продукт без ядра

- **API:** `/api/hypotheses` `[]`, `/api/decisions` `[]`, `/api/b2b` `[]`.
- **UI:** на Goals, Funnel и Снапшоте везде «0 сделок», «нет B2B», «Гипотез в работе: 0 + 0», «Решений в Decision Log: 0».
- **Симптом:** ядро методологии — Decision Log и гипотезы по формату Воронковой — не доступны через UI. В роутерах есть `POST /api/hypotheses`, `POST /api/decisions`, `POST /api/b2b`, но **в UI нет страниц** Hypotheses / Decisions / B2B-CRUD. В CLAUDE.md эти страницы описаны как обязательные (§Принципы, §Гипотеза без формата гипотезы — не гипотеза).
- **Должно быть:** добавить страницы /hypotheses, /decisions, /b2b с CRUD + валидацией формата (3 допущения, 2 метода, светофор, дедлайн, ICE=I×C×E).

#### C-010 — DOCX/PDF без оформления (форматирование «как туалетная бумага»)

- **Эталон:** страница /help — карточки с иконками, секциями, прозрачным фоном, выделениями.
- **Реальность отчёта (превью на /report и предположительно в DOCX/PDF):** только заголовки + плоский серый текст + тысячи дублирующих строк подряд без таблиц, без графиков, без цветных бейджей.
- **Должно быть (по требованию пользователя):**
  - Шапка с логотипом ProductCamp и периодом.
  - KPI-карточки (4–8 шт) в одну строку.
  - Каждый раздел оформлен как карточка с иконкой и подзаголовком.
  - **После каждого графика / таблицы — блок «Что хорошо / Что плохо»: зелёные пункты (✅) для хорошего, красные (🔴) для плохого.**
  - Графики встроены как изображения (ECharts → PNG, либо встроенные SVG через docx).
  - Таблицы с zebra-rows, выравниванием чисел справа, форматированием 1 234 / 1.2%.
  - В PDF — постраничная нумерация, оглавление с гиперссылками, footer с snapshotId.
  - В DOCX — стили заголовков Heading 1/2/3, TOC через `docx`-styles.
