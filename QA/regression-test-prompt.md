# QA — Регресс-промпт: полное тестирование функционала

> Роль: **QA-инженер**. Цель — за один прогон проверить весь функционал ProductCamp Conversion
> Analytics Tool и зафиксировать дефекты с приоритетами. Промпт самодостаточен: его можно отдать
> человеку или AI-агенту с доступом к репозиторию и запущенному приложению.
>
> 🔄 Актуально для **v0.9.0**: 11 страниц, авто-определение KPI-цели (`GOAL_ID=0`), детальный
> ≥25-страничный отчёт (DOCX/PDF/на экране), опциональный AI-анализ. Критерии приёмки новых фич —
> в [`../docs/specs/`](../docs/specs/README.md).

## Контекст продукта

Локальный инструмент (`./run.sh`): тянет данные Яндекс.Метрики (счётчик `<COUNTER_ID>`) в SQLite,
показывает дашборд (frontend `:5173`, API `:4000`), ведёт гипотезы по методологии Воронковой и
Decision Log, генерит детерминированные DOCX/PDF-отчёты и опциональный AI-анализ (Anthropic).

**Ключевые инварианты, которые НЕЛЬЗЯ нарушать (проверять в каждом сценарии):**

1. **Anti-hallucination:** любое число на дашборде/в отчёте прослеживается до `raw_responses` в SQLite
   (страница **Sources** по `raw_response_id`). Выдуманных чисел быть не должно.
2. **Заявка ≠ оплата:** B2C-заявки (goal reaches) и оплаченные B2B-билеты разделены везде.
3. **Детерминизм отчёта:** один `snapshotId` → идентичный контент DOCX/PDF (zip/pdf-таймстемпы —
   known limitation); в render-пути нет
   `Date.now()` и нет LLM-вызовов (AI-нарратив генерится заранее и хранится в снапшоте).
4. **Гипотеза без формата Воронковой не сохраняется** (≥3 допущения по 3 категориям, ≥2 метода,
   светофор, дедлайн, ICE = I×C×E).

## Подготовка окружения

| Шаг             | Команда                                                              | Ожидаемо                                                            |
| --------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Всё одной       | `./setup.sh`                                                         | install → init → start без ошибок                                   |
| Установка       | `pnpm install`                                                       | без ошибок                                                          |
| Инициализация   | `./init.sh`                                                          | `.env` заполнен (Anthropic key, OAuth, COUNTER_ID, GOAL_ID)         |
| Старт           | `./run.sh`                                                           | миграции → sync (или seed без токена) → дашборд открылся на `:5173` |
| Демо без токена | очистить `YANDEX_OAUTH_TOKEN`, `./run.sh`                            | дашборд наполнен `pnpm seed`                                        |
| Чистая БД       | `rm data/productcamp.sqlite*` → `pnpm --filter @pca/backend migrate` | 9 миграций применены                                                |

Прогнать автоматические проверки и зафиксировать результат:
`pnpm typecheck`, `pnpm lint`, `pnpm format:check`, `pnpm -r coverage` (порог 100%), `pnpm build`,
`pnpm e2e`. Любая красная — дефект **Blocker**.

**Реалии живых данных** (учитывать при проверке, это не дефекты):

- `GOAL_ID` может указывать на цель-**покупку** (напр. «Ecommerce: покупка») — тогда `goal_reaches` =
  оплаты B2C; уточните у владельца, что считается KPI «заявки». При `GOAL_ID=0` (по умолчанию) основная
  цель **определяется автоматически** (детерминированный классификатор по имени) — проверьте, что
  выбранная цель отражена в `SyncSummary.resolvedGoalId`, в `GET /api/metrics/primary-goal` и в бейдже
  «KPI-цель определена автоматически» на странице Overview.
- **Страницы выхода пусты на живых данных** — `ym:s:exitURL` не поддерживается Stat API (это ожидаемо).
- `bounceRate`/`conversionRate` хранятся как доля 0–1 (Метрика отдаёт проценты — нормализуются).

## Матрица функционала (проверить КАЖДЫЙ пункт)

### A. Парсер / синхронизация

- [ ] `pnpm auth` — печатает URL авторизации; после ввода кода пишет `YANDEX_OAUTH_TOKEN` в `.env`.
- [ ] неверный/просроченный код → понятная ошибка, токен не записан.
- [ ] `pnpm sync` без токена → корректно пропускает (`hasMetrikaToken` = false), не падает.
- [ ] `pnpm sync --goalId=<id> --from=YYYY-MM-DD --to=YYYY-MM-DD` → заполняет
      `channel_stats/utm_stats/geo_device_stats/page_stats`, `goal_reaches` > 0.
- [ ] период > 7 дней разбивается на дневные чанки; повторный sync дополняет историю, не теряя дни.
- [ ] недоступный атрибут (например `ym:s:exitURL`) → в лог `[sync] … skipped`, пайплайн НЕ падает,
      `exit_page_stats` = 0 строк.
- [ ] `bounceRate`/`conversionRate` хранятся как доля 0–1 (не проценты 0–100).
- [ ] каждый ответ API сохранён в `raw_responses` (проверить рост счётчика).

### B. API (`:4000`, Swagger `/docs`)

Проверить статусы и схемы: `GET /api/health` (флаг `metrikaTokenPresent`),
`GET /api/metrics/channels?from&to`, `/utm`, `/geo-device`, `/pages`, `/exit-pages`, `/goals?archived`,
`GET /api/metrics/raw/:id` (200/404), `GET /api/metrics/primary-goal` (200 авто-цель / 404 если KPI-цель
не найдена), `GET /api/hypotheses`, `/decisions`, `/b2b`,
`POST /api/sync`, `POST /api/hypotheses` (422 на невалидной по Воронковой),
`POST /api/decisions` (авто-обновление статуса гипотезы), `POST/PATCH/DELETE /api/b2b`,
`POST /api/report/snapshot`, `GET /api/report/snapshot/:id`, `POST /api/report/generate` (docx/pdf,
400 на неверном формате), `POST /api/report/insights` (200 / 404 нет снапшота / 503 нет ключа).

### C. Дашборд — постранично (11 страниц)

Для каждой: загрузка/ошибка/пустые данные; фильтры (период, сегмент, каналы); цифры совпадают с API.

- [ ] **Overview** — KPI (цель/заявки/gap), Daily-чарт, микс каналов, **«Слабые места»** (каналы с
      трафиком, но конверсией ниже средней; пустое состояние); бейдж **«KPI-цель определена
      автоматически: <имя>»** (скрыт, если `/api/metrics/primary-goal` отдаёт 404).
- [ ] **Traffic** — бар каналов, бейдж низкого UTM-покрытия, таблица UTM.
- [ ] **Audience** — бар «Топ стран», донат «Доля устройств», таблицы страна/устройство.
- [ ] **Behavior** — бары и таблицы «Страницы входа» (startURL) и «Страницы выхода» (на живых данных
      пустые — exitURL недоступен; UI не должен ломаться).
- [ ] **Trends** — линия визитов/заявок по дням + WoW-стрелки ▲/▼ (в т.ч. когда прошлой недели нет).
- [ ] **Funnel** — воронка Визиты → Заявки B2C → Билеты B2B → Оплачено B2B; поэтапная конверсия.
- [ ] **B2B** — CRUD сделок, этапы lead/negotiation/invoiced/paid, прогноз; «Добавить» с валидацией.
- [ ] **Hypotheses** — список по ICE; редактор **блокирует сохранение** без полного формата Воронковой.
- [ ] **Decisions** — Decision Log; создание решения авто-обновляет статус связанной гипотезы.
- [ ] **Report** — «Сформировать snapshot», **полный отчёт на экране** (ReportFullView, идентичен
      DOCX/PDF), Export DOCX, Export PDF, кнопка AI-анализа.
- [ ] **Sources** — поиск сырого ответа по `raw_response_id`; показывает endpoint/query_hash/период/payload.

### D. Отчёты (детерминизм + содержание)

- [ ] DOCX и PDF из одного `snapshotId` → одинаковый контент (секции: cover, exec summary, methodology
      с атрибуцией Воронковой, воронка визит→заявка→оплата, анализ по каналам, приоритизация по ICE,
      define/develop гипотезы — каждая раскрыта полностью (формат Воронковой, допущения по категориям,
      методы, разбор ICE, светофор, дедлайн, статус), deliver/Decision Log, AI-анализ если сгенерирован,
      топ UTM/гео/страниц, дорожная карта, глоссарий, data appendix).
- [ ] отчёт **детальный (≥25 страниц A4)**, каждая секция с новой страницы; на демо-сидах → ~28 стр. PDF.
- [ ] PDF требует `PUPPETEER_EXECUTABLE_PATH`; без него — понятная ошибка, DOCX работает.
- [ ] числа в отчёте = числа на дашборде = `raw_responses`.

### E. AI-анализ (Anthropic)

- [ ] без `ANTHROPIC_API_KEY` → `POST /api/report/insights` отдаёт 503 с понятным текстом; дашборд жив.
- [ ] с ключом → нарратив генерится, помечен как AI-интерпретация, сохраняется в снапшот.
- [ ] нарратив опирается ТОЛЬКО на числа снапшота (проверить отсутствие выдуманных фактов).
- [ ] после генерации DOCX/PDF содержат секцию «AI-анализ» (тот же текст, без повторного вызова LLM).

### F. Граничные случаи и устойчивость

- [ ] пустой период / нет данных → пустые состояния, без падений.
- [ ] очень длинные URL/названия каналов → не ломают таблицы/графики (URL сокращаются до пути).
- [ ] архивные цели скрыты по умолчанию, тогл показывает.
- [ ] повторная инициализация `./init.sh` не затирает чужие переменные `.env` (только нужные ключи).
- [ ] перезапуск `./run.sh` идемпотентен (миграции не дублируются).

## Формат отчёта о дефектах

Для каждого дефекта: **ID**, шаги воспроизведения, ожидаемо/фактически, **приоритет**
(Blocker / Critical / Major / Minor / Trivial), скриншот/лог, нарушенный инвариант (если есть),
ссылка на страницу/эндпоинт. В конце — сводка: сколько пройдено/провалено по разделам A–F и вердикт
go/no-go для использования командой трека.
