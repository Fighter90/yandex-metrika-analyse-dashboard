## Контекст продукта

Локальный инструмент (`./run.sh`): тянет данные Яндекс.Метрики (счётчик `<COUNTER_ID>`) в SQLite,
показывает дашборд (frontend `:5173`, API `:4000`), ведёт гипотезы по методологии проверки гипотез
и Decision Log, генерит детерминированные DOCX/PDF-отчёты и опциональный AI-анализ (Anthropic).

**Ключевые инварианты, которые НЕЛЬЗЯ нарушать (проверять в каждом сценарии):**

0. **Согласованность чисел между страницами:** одна и та же метрика (визиты, заявки, CR) за один и тот же период должна совпадать на Overview / Traffic / Behavior / Funnel / Goals / Snapshot / DOCX / PDF. Несовпадение = Blocker (нарушение anti-hallucination).
1. **Anti-hallucination:** любое число на дашборде/в отчёте прослеживается до `raw_responses` в SQLite. Выдуманных чисел быть не должно.
2. **Заявка ≠ оплата:** B2C-заявки (goal reaches) и оплаченные B2B-билеты разделены везде.
3. **Детерминизм отчёта:** один `snapshotId` → идентичный контент DOCX/PDF (zip/pdf-таймстемпы — known limitation); в render-пути нет `Date.now()` и нет LLM-вызовов (AI-нарратив генерится заранее и хранится в снапшоте).
4. **Гипотеза без полного формата методологии не сохраняется** (≥3 допущения по 3 категориям, ≥2 метода, светофор, дедлайн, ICE = I×C×E).
5. **Фильтры не более 1 года:** кастомный date picker ограничивает период 365 днями, пресеты 7д/14д/30д/90д/1г.

## Подготовка окружения

| Шаг             | Команда                                                              | Ожидаемо                                                            |
| --------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Всё одной       | `./setup.sh`                                                         | install → init → start без ошибок                                   |
| Установка       | `pnpm install`                                                       | без ошибок                                                          |
| Инициализация   | `./init.sh`                                                          | `.env` заполнен (Anthropic key, OAuth, COUNTER_ID, GOAL_ID)         |
| Старт           | `./run.sh`                                                           | миграции → sync (или seed без токена) → дашборд открылся на `:5173` |
| Демо без токена | очистить `YANDEX_OAUTH_TOKEN`, `./run.sh`                            | дашборд наполнен `pnpm seed`                                        |
| Чистая БД       | `rm data/productcamp.sqlite*` → `pnpm --filter @pca/backend migrate` | 9 миграций применены                                                |

Прогнать автоматические проверки и зафиксировать результат:
`pnpm typecheck`, `pnpm lint`, `pnpm format:check`, `pnpm -r coverage` (порог 100%), `pnpm build`,
`pnpm e2e`. Любая красная — дефект **Blocker**.

**Реалии живых данных** (учитывать при проверке, это не дефекты):

- `GOAL_ID` может указывать на цель-**покупку** (напр. «Ecommerce: покупка») — тогда `goal_reaches` = оплаты B2C. При `GOAL_ID=0` основная цель определяется автоматически.
- **Страницы выхода пусты на живых данных** — `ym:s:exitURL` не поддерживается Stat API.
- `bounceRate`/`conversionRate` хранятся как доля 0–1 (Метрика отдаёт проценты — нормализуются).
- COUNTER_ID должен быть корректным (задаётся в `.env`), иначе sync не работает.

## Матрица функционала (проверить КАЖДЫЙ пункт)

### A. Парсер / синхронизация

- [ ] `pnpm auth` — печатает URL авторизации; после ввода кода пишет `YANDEX_OAUTH_TOKEN` в `.env`.
- [ ] неверный/просроченный код → понятная ошибка, токен не записан.
- [ ] `pnpm sync` без токена → корректно пропускает (`hasMetrikaToken` = false), не падает.
- [ ] `pnpm sync --goalId=<id> --from=YYYY-MM-DD --to=YYYY-MM-DD` → заполняет каналы/UTM/гео/страницы.
- [ ] период > 7 дней разбивается на дневные чанки; повторный sync дополняет историю, не теряя дни.
- [ ] недоступный атрибут → в лог `[sync] … skipped`, пайплайн НЕ падает.
- [ ] `bounceRate`/`conversionRate` хранятся как доля 0–1.
- [ ] каждый ответ API сохранён в `raw_responses`.
- [ ] **Сверка с Метрикой по дням:** за любой день `SUM(visits)`/`SUM(users)` в `channel_stats` совпадают с `totals`/строками сырого ответа traffic-by-source; на странице Метрики за тот же период (модель **LastSign**, без cross-device) визиты совпадают.
- [ ] **`attribution=lastsign`** присутствует в параметрах всех запросов Stat API.
- [ ] запрос трафика использует **одно** измерение `ym:s:lastTrafficSource` (без `lastSourceEngine`) — одна строка на канал/день; повторный sync с неверным/просроченным токеном **НЕ обнуляет** БД (валидация целей до `resetSyncedData`).

## Формат отчёта о дефектах

Для каждого дефекта: **ID**, шаги воспроизведения, ожидаемо/фактически, **приоритет** (Blocker / Critical / Major / Minor / Trivial), скриншот/лог, нарушенный инвариант, ссылка на страницу/эндпоинт. В конце — сводка: сколько пройдено/провалено по разделам A–J и вердикт go/no-go.
