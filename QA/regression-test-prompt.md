## Контекст продукта

Локальный инструмент (`./run.sh`): тянет данные Яндекс.Метрики (счётчик `<COUNTER_ID>`) в SQLite,
показывает дашборд (frontend `:5173`, API `:4000`), ведёт гипотезы по методологии Воронковой и
Decision Log, генерит детерминированные DOCX/PDF-отчёты и опциональный AI-анализ (Anthropic).

**Ключевые инварианты, которые НЕЛЬЗЯ нарушать (проверять в каждом сценарии):**

0. **Согласованность чисел между страницами:** одна и та же метрика (визиты, заявки, CR) за один и тот же период должна совпадать на Overview / Traffic / Behavior / Funnel / Goals / Snapshot / DOCX / PDF. Несовпадение = Blocker (нарушение anti-hallucination).
1. **Anti-hallucination:** любое число на дашборде/в отчёте прослеживается до `raw_responses` в SQLite. Выдуманных чисел быть не должно.
2. **Заявка ≠ оплата:** B2C-заявки (goal reaches) и оплаченные B2B-билеты разделены везде.
3. **Детерминизм отчёта:** один `snapshotId` → идентичный контент DOCX/PDF (zip/pdf-таймстемпы — known limitation); в render-пути нет `Date.now()` и нет LLM-вызовов (AI-нарратив генерится заранее и хранится в снапшоте).
4. **Гипотеза без формата Воронковой не сохраняется** (≥3 допущения по 3 категориям, ≥2 метода, светофор, дедлайн, ICE = I×C×E).
5. **Фильтры не более 1 года:** кастомный date picker ограничивает период 365 днями, пресеты 7д/14д/30д/90д/1г.

## Подготовка окружения

| Шаг             | Команда                                                              | Ожидаемо                                                            |
| --------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Всё одной       | `./setup.sh`                                                         | install → init → start без ошибок                                   |
| Установка       | `pnpm install`                                                       | без ошибок                                                          |
| Инициализация   | `./init.sh`                                                          | `.env` заполнен (Anthropic key, OAuth, COUNTER_ID, GOAL_ID)         |
| Старт           | `./run.sh`                                                           | миграции → sync (или seed без токена) → дашборд открылся на `:5173` |
| Демо без токена | очистить `YANDEX_OAUTH_TOKEN`, `./run.sh`                            | дашборд наполнен `pnpm seed`                                        |
| Чистая БД       | `rm data/productcamp.sqlite*` → `pnpm --filter @pca/backend migrate` | 9 миграций применены                                                |

Прогнать автоматические проверки и зафиксировать результат:
`pnpm typecheck`, `pnpm lint`, `pnpm format:check`, `pnpm -r coverage` (порог 100%), `pnpm build`,
`pnpm e2e`. Любая красная — дефект **Blocker**.

**Реалии живых данных** (учитывать при проверке, это не дефекты):

- `GOAL_ID` может указывать на цель-**покупку** (напр. «Ecommerce: покупка») — тогда `goal_reaches` = оплаты B2C. При `GOAL_ID=0` основная цель определяется автоматически.
- **Страницы выхода пусты на живых данных** — `ym:s:exitURL` не поддерживается Stat API.
- `bounceRate`/`conversionRate` хранятся как доля 0–1 (Метрика отдаёт проценты — нормализуются).
- COUNTER_ID должен быть корректным (напр. 54280963), иначе sync не работает.

## Матрица функционала (проверить КАЖДЫЙ пункт)

### A. Парсер / синхронизация

- [ ] `pnpm auth` — печатает URL авторизации; после ввода кода пишет `YANDEX_OAUTH_TOKEN` в `.env`.
- [ ] неверный/просроченный код → понятная ошибка, токен не записан.
- [ ] `pnpm sync` без токена → корректно пропускает (`hasMetrikaToken` = false), не падает.
- [ ] `pnpm sync --goalId=<id> --from=YYYY-MM-DD --to=YYYY-MM-DD` → заполняет каналы/UTM/гео/страницы.
- [ ] период > 7 дней разбивается на дневные чанки; повторный sync дополняет историю, не теряя дни.
- [ ] недоступный атрибут → в лог `[sync] … skipped`, пайплайн НЕ падает.
- [ ] `bounceRate`/`conversionRate` хранятся как доля 0–1.
- [ ] каждый ответ API сохранён в `raw_responses`.

### B. API (`:4000`, Swagger `/docs`)

Проверить статусы и схемы: `GET /api/health`, `/metrics/channels`, `/metrics/utm`, `/metrics/geo-device`, `/metrics/pages`, `/metrics/exit-pages`, `/metrics/goals?archived`, `/metrics/primary-goal`, `/metrics/raw/:id`, `/hypotheses`, `/decisions`, `/b2b`, `/report/snapshot`, `/report/snapshot/:id`, `/report/generate`, `/report/insights`, `/report/hypotheses`, `/report/snapshots`, `/settings`, `/settings/refresh`, `/sync`.

### C. Дашборд — постранично (9 страниц)

Для каждой: загрузка/ошибка/пустые данные; фильтры; цифры совпадают с API.

- [ ] **Overview** — KPI (цель/заявки/gap), график визитов+заявок по дням, микс каналов, топ стран, доля устройств, UTM-таблица, страницы входа/выхода, слабые места, цветные бейджи. **FilterBar виден.**
- [ ] **Traffic** — бар каналов, визиты vs заявки, таблица каналов с CR, UTM-разбивка, insights. **FilterBar виден.**
- [ ] **Behavior** — CR страниц входа, отказы страниц входа/выхода, таблицы с подсветкой, рекомендации. **FilterBar виден.**
- [ ] **Funnel** — 4 этапа (Визиты → Заявки B2C → B2B pipeline → Оплачено B2B), анализ потерь, CR по каналам, B2B по этапам, проблемные страницы. **FilterBar виден.**
- [ ] **Goals** — прогресс-ринг к 300 билетам, метрики, B2B сделки, рекомендации. **FilterBar виден.**
- [ ] **Report** — генерация снапшота, AI-анализ (5 секций, HTML-рендеринг), экспорт DOCX/PDF, перестроение. **FilterBar виден.**
- [ ] **History** — список снапшотов, кнопка «Просмотреть» → открывает сохранённый отчёт без перегенерации. **FilterBar виден.**
- [ ] **Settings** — OAuth-токен, Client ID/Secret, COUNTER_ID (показывает текущий), GOAL_ID select из Метрики (активные + архивные), ANTHROPIC_API_KEY, синхронизация с прогресс-баром (10 стадий). **FilterBar скрыт.**
- [ ] **Help** — полная документация: описание всех страниц, фильтры, FAQ (10 вопросов), глоссарий. **FilterBar скрыт.**
- [ ] **Hypotheses (после v2.4.0)** — список + форма создания гипотезы по Воронковой; форма блокирует Save без 3 допущений, 2 методов, светофора, дедлайна, ICE=I×C×E.
- [ ] **Decisions (после v2.4.0)** — Decision Log; каждая запись связана со снапшотом и гипотезой; экспорт CSV.
- [ ] **B2B (после v2.4.0)** — CRUD `b2b_manual`: компания, стадия, билетов, ожидаемая оплата, ответственный.

### D. Фильтры (шапка дашборда)

- [ ] Пресеты: 7д, 14д, 30д, 90д, 1г — переключают период.
- [ ] Кастомный date picker (от/до) — применяет произвольный период.
- [ ] Ограничение макс. 365 дней — при попытке выбрать больше показывается ошибка.
- [ ] Сегмент (B2C / B2C+B2B / B2B) — фильтрует каналы и UTM.
- [ ] Архивные цели — checkbox включает/отключает показ архивных целей.
- [ ] FilterBar скрыт на страницах /help и /settings.
- [ ] Мобильное меню (гамбургер) на экранах < 1024px.

### E. Отчёты (детерминизм + содержание + ГОСТ-форматирование)

- [ ] DOCX и PDF из одного `snapshotId` → одинаковый контент (SHA-256 совпадает).
- [ ] отчёт содержит: титульный лист, реферат, содержание (TOC), введение, разделы (KPI, трафик, UTM, гео+девайс, страницы, воронка, B2B, AI-анализ, гипотезы, DL), заключение, источники, приложения.
- [ ] **в снапшоте нет дублей** (`SELECT COUNT(*) FROM channel_stats GROUP BY date, channel HAVING COUNT(*) > 1` = 0).
- [ ] **счётчик «Каналов: N»** в превью соответствует реальному количеству каналов × дней (не «2293»).
- [ ] **После каждого графика/таблицы** — блок «🟢 Что хорошо» (зелёный фон) и «🔴 Что плохо» (красный фон) с осмысленными пунктами на основе данных.
- [ ] **Формат ГОСТ Р 7.32-2017:** шрифт Arial/Times New Roman 11–12pt, line-spacing 1.5, поля 30/15/20/20мм, нумерация страниц, подписи рисунков/таблиц («Рисунок N — …», «Таблица N — …»).
- [ ] Графики встроены как изображения (не текст), графики не «обрезаны до 3 каналов из 7».
- [ ] AI-анализ рендерится как HTML (таблицы, заголовки, форматирование).
- [ ] PDF авто-находит Chrome на macOS/Linux/Windows.
- [ ] числа в отчёте = числа на дашборде = `raw_responses`; **визиты совпадают на всех страницах**.

### F. AI-анализ (Anthropic)

- [ ] без `ANTHROPIC_API_KEY` → `POST /api/report/insights` отдаёт 503; дашборд жив.
- [ ] с ключом → нарратив генерится в 5 секций (~30–60с), сохраняется в снапшот.
- [ ] нарратив опирается ТОЛЬКО на числа снапшота.
- [ ] после генерации DOCX/PDF содержат секцию «AI-анализ».
- [ ] прогресс-бар показывает текущую стадию генерации.

### G. Settings (настройки)

- [ ] GET /api/settings возвращает COUNTER_ID из .env (не маскируется).
- [ ] Маскированные секреты (\*\*\*\*) не отправляются обратно на сервер при сохранении.
- [ ] COUNTER_ID отправляется только если пользователь явно его изменил.
- [ ] GOAL_ID select показывает все цели из Метрики (активные + архивные).
- [ ] Прогресс-бар синхронизации показывает 10 стадий с процентом и описанием.
- [ ] Текущий счётчик отображается из health API если settings возвращает 0.

### H. Граничные случаи и устойчивость

- [ ] пустой период / нет данных → пустые состояния, без падений.
- [ ] очень длинные URL/названия → не ломают таблицы/графики.
- [ ] архивные цели скрыты по умолчанию, тогл показывает.
- [ ] повторная инициализация `./init.sh` не затирает чужие переменные `.env`.
- [ ] перезапуск `./run.sh` идемпотентен.
- [ ] COUNTER_ID=0 → демо-режим с seed-данными.
- [ ] Дата окончания раньше даты начала → ошибка в date picker.

## Формат отчёта о дефектах

Для каждого дефекта: **ID**, шаги воспроизведения, ожидаемо/фактически, **приоритет** (Blocker / Critical / Major / Minor / Trivial), скриншот/лог, нарушенный инвариант, ссылка на страницу/эндпоинт. В конце — сводка: сколько пройдено/провалено по разделам A–H и вердикт go/no-go.
