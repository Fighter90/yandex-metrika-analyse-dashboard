# Skill — Hypothesis Check (ProductCamp adaptation)

> Структурирует сырую гипотезу + считает ICE.

---

## Назначение

Превращает «у меня есть идея» в проверяемую гипотезу с критериями успеха и приоритетом.

В контексте ProductCamp — формулируем **гипотезы проблем** (почему не покупают / отваливаются) и **гипотезы решений** (как починить) для трека «Конверсии и лидген».

## Промпт

```
Я хочу проверить продуктовую гипотезу в рамках трека «Конверсии и лидген» ProductCamp.
Контекст — KPI 300 платных билетов, осталось ~2 недели, факт ~150.

Сырая идея: {опишите идею своими словами}

Сделай следующее:

1. Переформулируй в формат:
   «{ЦА} {готова сделать что-то / готова заплатить за / переключится на} {решение}, если {условие}»

2. Найди скрытые допущения (минимум 3, по одному из категорий):
   - Поведение ЦА (что мы предполагаем о действиях посетителей/B2B-клиентов)
   - Рынок (что мы предполагаем о конкурентах/трендах/готовности платить)
   - Технология/канал (что мы предполагаем о работоспособности канала, UTM, посадочной)

3. Определи как проверить (минимум 2 способа):
   - Синтетический CustDev → какие вопросы задать (см. .qwen/skills/synthetic-custdev/SKILL.md)
   - Живой CustDev → сколько респондентов и какие критерии
   - Количественный → какой запрос к данным Метрики (UTM-source, goal_id, период, порог)
   - Рыночный → какие сигналы искать (см. .qwen/skills/market-scan/SKILL.md)

4. Оцени по ICE (1–10), обоснуй каждую цифру в одну строку:
   - Impact: насколько изменит количество платных билетов до старта кемпа, если подтвердится
   - Confidence: насколько уверен, что подтвердится (есть ли уже сигналы)
   - Ease: насколько легко проверить за оставшееся время

   ICE считается как ПРОИЗВЕДЕНИЕ: I × C × E (диапазон 1–1000).

5. Дай критерии светофора:
   - 🟢 Зелёный (продолжаем, внедряем): конкретный порог с метрикой
   - 🟡 Жёлтый (pivot): что меняем
   - 🔴 Красный (стоп): почему окончательно

6. Предложи дедлайн проверки (в днях). Помни — до старта кемпа осталось мало.
```

## Что НЕ делать

- Не запускайте проверку без чёткого критерия успеха.
- Не доверяйте ICE от AI без своей корректировки — это draft, не вердикт.
- Не сравнивайте ICE между гипотезами разного масштаба (problem vs solution).
- Не используйте средний ICE — у нас формула product (см. `docs/methodology-ice.md`).

## Пример заполнения 1 — Problem Hypothesis

| Поле | Значение |
|---|---|
| Сырая идея | Подкаст приводит много трафика, но мало платных |
| Структура | Слушатель подкаста ProductCamp **не покупает** билет за 14k+, **если** между прослушиванием и landing'ом нет промежуточного экрана с конкретной программой и спикерами |
| Допущение 1 (поведение) | Слушатель подкаста — холодный лид, требует прогрева |
| Допущение 2 (рынок) | На рынке конференций 14k+ воспринимается как «дорого» без чёткой ценности |
| Допущение 3 (канал) | UTM подкаста корректно проставлены за последние 7 дней |
| Проверка количественная | SQL/Метрика: conversion rate (визит → оплата) для `utm_source=podcast` vs остальных за 14 дней |
| Проверка синтетическая | CustDev с симулированным «продактом-слушателем подкаста, услышал 2 эпизода» |
| ICE | I=7 (если подтвердится — есть кого активировать), C=6 (есть подозрение, нет данных), E=9 (один SQL-запрос) → **378** |
| 🟢 Зелёный | conversion rate подкаст-канала на 30%+ ниже среднего по другим каналам |
| 🟡 Жёлтый | разница 10–30% — нужен глубже look |
| 🔴 Красный | разницы нет / подкаст конвертит лучше среднего — гипотеза опровергнута |
| Дедлайн | 2 дня |

## Пример заполнения 2 — Solution Hypothesis (привязанная к Problem)

| Поле | Значение |
|---|---|
| Связь с problem | Подтверждённая гипотеза проблемы DL-005 (подкаст → низкая конверсия) |
| Сырая идея | Сделать промежуточный лендинг для подкаст-аудитории |
| Структура | Слушатель подкаста **переключится** с просто landing'а на промежуточный экран «специально для слушателей подкаста» с конкретной программой 3 ключевых докладов, **если** ссылка в подкаст-описании ведёт сразу туда (utm_content=podcast_landing) |
| Допущение 1 (поведение) | Релевантная программа закрывает возражение «дорого без понимания ценности» |
| Допущение 2 (рынок) | 2 ключевых конкурента используют похожий приём, значит — паттерн работает |
| Допущение 3 (тех) | Можно сделать страницу за 4 часа на конструкторе, без редизайна |
| Проверка | A/B-сплит на UTM-уровне (50/50) на 5 дней; min sample 200 визитов на каждую ветку |
| ICE | I=8 (если +30% conversion на подкаст → +15 билетов), C=4 (не проверяли A/B такого формата ранее), E=6 (4 часа на лендинг + правка ссылок в подкастах) → **192** |
| 🟢 Зелёный | conversion rate новой ветки выше старой на ≥25% при p<0.1 |
| 🟡 Жёлтый | улучшение 10–25% — оставляем, не масштабируем |
| 🔴 Красный | разницы нет или хуже — откатываем |
| Дедлайн | 7 дней |

## Чек-лист перед сохранением

- [ ] Формат «{subject} {action} {solution}, если {condition}» — все 4 части заполнены
- [ ] ≥3 допущения с категориями (поведение / рынок / тех или канал)
- [ ] ≥2 метода проверки с конкретными планами
- [ ] ICE с rationale для каждого из I, C, E
- [ ] Светофор с конкретными порогами и метриками
- [ ] Дедлайн в днях, реалистичный к старту ProductCamp

Если хотя бы один пункт не покрыт — гипотеза не готова. UI инструмента это блокирует автоматически.
