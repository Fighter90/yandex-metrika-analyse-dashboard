# Руководство пользователя — ProductCamp Conversion Analytics Dashboard

> **Текущая версия:** v2.3.0 (2026-05-27)

## О системе

ProductCamp Conversion Analytics Dashboard — это аналитический инструмент для трека «Конверсии и лидген» ProductCamp. Система подключается к Яндекс.Метрике по OAuth, загружает данные, сохраняет их в локальной базе данных SQLite и предоставляет интерактивный дашборд для анализа конверсий.

**Основная цель:** отслеживать прогресс к KPI в 300+ платных билетов и анализировать, какие каналы трафика приносят заявки, а какие — пустые визиты.

**Принцип:** заявка ≠ оплата. Заявки B2C — это достижения цели в Метрике. Оплаты — это фактически оплаченные билеты (B2B вводится вручную).

### Что нового в v2.3.0

- **AI-анализ без обрезаний**: полный рендеринг с таблицами, списками, заголовками
- **DOCX/PDF экспорт**: GOST форматирование (Times New Roman 14pt, межстрочный 1.5)
- **Gap считается по оплатам**: 300 - оплаченные B2B билеты, не заявки
- **История → «Просмотреть»**: загружает сохранённый AI-анализ без повторной генерации
- **GOAL_ID select**: выбор цели из всех целей Метрики
- **Фильтры до 1 года**: пресеты 7д/14д/30д/90д/1г, кастомный date picker (макс. 365 дней)

## 1. Развёртывание системы

### 1.1 Требования

- Node.js 20+ (проверка: `node --version`)
- pnpm 9+ (проверка: `pnpm --version`)
- Git
- Доступ к Яндекс.Метрике (счётчик с правом `metrika:read`)

### 1.2 Установка

```bash
# Клонировать репозиторий
git clone git@github.com:Fighter90/metrika_analyse_dashboard.git
cd metrika_analyse_dashboard

# Установить зависимости
pnpm install
```

### 1.3 Инициализация

```bash
./init.sh
```

Скрипт интерактивно создаёт файл `.env` и запрашивает:

1. **ANTHROPIC_API_KEY** — ключ для AI-анализа отчётов (можно пропустить, дашборд работает без него)
2. **COUNTER_ID** — ID счётчика Яндекс.Метрики (например, 54280963 для ProductCamp)
3. **GOAL_ID** — ID цели KPI (0 = авто-определение, >0 = зафиксировать конкретную цель)
4. Предложение настроить OAuth Яндекс.Метрики (`pnpm auth`)

### 1.4 Настройка OAuth

Для загрузки живых данных из Метрики нужен OAuth-токен:

```bash
pnpm auth
```

Скрипт:
1. Выводит ссылку для авторизации
2. Просит скопировать код подтверждения из браузера
3. Обменивает код на токен и записывает в `.env`

### 1.5 Запуск

```bash
./run.sh
```

Скрипт выполняет:
1. Установку pnpm (если отсутствует)
2. Миграции базы данных
3. Синхронизацию данных из Метрики (если есть токен) или загрузку демо-данных
4. Запуск backend (порт 4000) и frontend (порт 5173)
5. Открытие дашборда в браузере

**URL:**
- Дашборд: http://localhost:5173
- API: http://localhost:4000
- Swagger API docs: http://localhost:4000/docs

## 2. Страницы дашборда

### 2.1 Обзор (/)

Главная страница с общей картиной:

- **KPI-стрип:** цель (300), заявки B2C, gap до цели
- **График визитов и заявок по дням** — динамика за выбранный период
- **Микс каналов** — круговая диаграмма распределения трафика
- **Топ стран** — горизонтальный бар-чарт
- **Доля устройств** — donut-чарт (смартфоны, ПК, планшеты)
- **UTM-разбивка** — таблица с визитами, заявками, CR по каждой кампании
- **Страницы входа/выхода** — топ-5 страниц с bounce rate
- **Слабые места** — каналы с CR ниже среднего

**Цветовая индикация:**
- 🟢 Зелёный: CR > 2% — хорошо, масштабировать
- 🟡 Жёлтый: CR 1–2% — нормально, есть потенциал
- 🔴 Красный: CR < 1% — проблема, проверить качество трафика

### 2.2 Трафик (/traffic)

Детальный анализ каналов:

- **Бар-чарт визитов по каналам**
- **Визиты vs заявки** — какой канал лучше конвертирует
- **Таблица каналов** — визиты, пользователи, заявки, CR
- **UTM-разбивка** — детальная таблица по source/medium/campaign

**Рекомендации:**
- Каналы с CR выше среднего → масштабировать
- Каналы с CR < 0.5% и большим объёмом → проверить качество
- UTM-покрытие < 70% → настроить разметку ссылок

### 2.3 Поведение (/behavior)

Анализ посадочных страниц:

- **Конверсия страниц входа** — горизонтальный бар-чарт с цветовой индикацией
- **Отказы страниц входа** — какие страницы теряют больше всего пользователей
- **Отказы страниц выхода** — точки отвала в воронке
- **Таблицы** — URL, визиты, отказы, заявки, CR с подсветкой

**Рекомендации:**
- Страницы с bounce > 40% → упростить контент
- Страницы с высоким трафиком и низким CR → добавить CTA
- Страницы с низким bounce → использовать как референс

### 2.4 Воронка (/funnel)

Воронка конверсии:

- **4 этапа:** Визиты → Заявки B2C → B2B pipeline → Оплачено B2B
- **Анализ потерь** — сколько пользователей потеряно на каждом этапе
- **CR по каналам** — какой канал лучше проходит воронку
- **B2B по этапам** — lead, negotiation, invoiced, paid
- **Проблемные страницы** — высокий bounce, низкий CR

**Рекомендации:**
- Переход заявка → оплата близок к 0% → настроить follow-up (авто-письмо + звонок)
- B2B pipeline пуст → запустить активные продажи
- Мобильный CR ниже десктопного → проверить мобильную версию

### 2.5 Цели (/goals)

Прогресс к цели в 300 платных билетов:

- **Прогресс-ринг** — визуализация текущего прогресса
- **Метрики:** визиты, общий CR, B2B сделки, процент прогресса
- **B2B сделки** — таблица всех сделок с этапами и суммами
- **Рекомендации** — конкретные действия на основе текущих данных

**Как считается прогресс:**
B2B оплачено (из ручного пайплайна) + ~30% от заявок B2C (оценка конверсии заявка→оплата)

### 2.6 Отчёт (/report)

Генерация детального отчёта:

1. Выберите период в шапке дашборда
2. Нажмите **«Сформировать срез данных»**
3. (Опц.) Нажмите **«Сгенерировать AI-анализ»** (нужен ANTHROPIC_API_KEY)
4. Нажмите **«Перестроить отчёт»** для обновления
5. Экспортируйте в **DOCX** или **PDF**

**AI-анализ:**
- Генерируется в 5 секций: Итог, Каналы/UTM/Аудитория, Страницы/Воронка, Риски/Рекомендации, Гипотезы/Дорожная карта
- Каждая секция — отдельный запрос к Anthropic Claude
- Результат рендерится как HTML с форматированием (таблицы, заголовки)
- Время генерации: 30–60 секунд

### 2.7 История (/history)

Список всех сгенерированных отчётов:

- **ID** — уникальный идентификатор снапшота
- **Сформирован** — дата и время создания
- **Период** — даты, за которые собран отчёт
- **Просмотреть** — открывает сохранённый отчёт без перегенерации

### 2.8 Настройки (/settings)

Подключение к Яндекс.Метрике:

- **Текущий счётчик** — отображает текущий COUNTER_ID (из .env или health API)
- **OAuth-токен** — токен с правом metrika:read
- **Client ID/Secret** — для OAuth-аутентификации
- **Счётчик Метрики** — ID счётчика (54280963 для ProductCamp)
- **Цель KPI (GOAL_ID)** — выпадающий список всех целей из Метрики
  - 0 = авто-определение
  - Выберите ID = зафиксировать конкретную цель
  - Показывает активные и архивные цели с названиями
- **ANTHROPIC_API_KEY** — ключ для AI-анализа
- **🔄 Обновить данные из Метрики** — синхронизация последних 14 дней
  - Прогресс-бар с 10 стадиями:
    1. Подключение к Метрике
    2. Загрузка целей
    3. Загрузка каналов
    4. Загрузка UTM
    5. Загрузка гео
    6. Загрузка страниц входа
    7. Загрузка страниц выхода
    8. Сохранение в БД
    9. Обновление отчётов
    10. Готово!

### 2.9 Справка (/help)

Полная документация по дашборду:

- Описание каждой страницы и графиков
- Описание фильтров (пресеты, даты, сегмент, архивные цели)
- FAQ (10 вопросов)
- Глоссарий терминов (CR, bounce rate, UTM, B2C, B2B, ICE и др.)

## 3. Фильтры (шапка дашборда)

Панель фильтров находится вверху каждой страницы и влияет на все графики:

- **7д / 14д / 30д** — быстрые пресеты периода
- **📅 Даты** — кастомный период (дата начала → дата окончания → Применить)
- **Сегмент:**
  - **B2C** — только потребительские каналы (Direct, Search, Social и др.)
  - **B2C+B2B** — все каналы (по умолчанию)
  - **B2B** — только B2B-каналы (если настроены отдельно)
- **Архивные цели** — показать/скрыть цели Метрики, которые были удалены или заархивированы

## 4. Источники данных

### 4.1 Яндекс.Метрика

Данные загружаются через Stat API:

| Эндпоинт | Данные |
|----------|--------|
| `traffic-by-source` | Визиты, пользователи, отказы, заявки, CR по каналам |
| `utm-breakdown` | Разбивка по UTM: source, medium, campaign |
| `geo-device-breakdown` | Гео (страна) + устройство (смартфон, ПК, планшет) |
| `page-behavior` (startURL) | Страницы входа: визиты, отказы, заявки |
| `page-behavior` (exitURL) | Страницы выхода: визиты, отказы, заявки |

### 4.2 Ручной ввод (B2B)

B2B-сделки вводятся вручную через API `POST /api/b2b`:

| Поле | Описание |
|------|----------|
| `company` | Название компании |
| `tickets` | Количество билетов |
| `stage` | Этап: lead, negotiation, invoiced, paid |
| `amountRub` | Сумма в рублях (опционально) |
| `contactEmail` | Email контакта (опционально) |
| `notes` | Заметки (опционально) |
| `dateAdded` | Дата добавления |
| `datePaid` | Дата оплаты (опционально) |

### 4.3 База данных

SQLite хранит:

| Таблица | Описание |
|---------|----------|
| `channel_stats` | Данные по каналам |
| `utm_stats` | UTM-разбивка |
| `geo_device_stats` | Гео + устройства |
| `page_stats` | Страницы входа |
| `exit_page_stats` | Страницы выхода |
| `raw_responses` | Сырые ответы Метрики (прослеживаемость) |
| `b2b_manual` | Ручные B2B-сделки |
| `snapshots` | Срезы данных для отчётов |

## 5. Устранение проблем

### 5.1 Дашборд не запускается

```bash
# Проверить Node.js
node --version  # должен быть >= 20

# Проверить pnpm
pnpm --version  # должен быть >= 9

# Переустановить зависимости
rm -rf node_modules
pnpm install
```

### 5.2 Нет данных из Метрики

1. Проверьте COUNTER_ID в настройках (должен быть > 0)
2. Проверьте YANDEX_OAUTH_TOKEN в `.env` (должен быть валидным)
3. Нажмите «Обновить данные из Метрики» в настройках
4. Проверьте логи backend: `pnpm --filter @pca/backend dev`

### 5.3 PDF экспорт не работает

Ошибка: `An executablePath or channel must be specified for puppeteer-core`

**Решение:** Укажите путь к Chrome в `.env`:

```
PUPPETEER_EXECUTABLE_PATH=/Applications/Google Chrome.app/Contents/MacOS/Google Chrome
```

Или установите Chrome: https://www.google.com/chrome/

### 5.4 AI-анализ не генерируется

1. Проверьте ANTHROPIC_API_KEY в настройках
2. Проверьте подключение к интернету
3. Попробуйте снова через 30 секунд

### 5.5 GOAL_ID не определяется автоматически

1. Убедитесь, что в Метрике есть цели с типом «покупка» или «оплата»
2. Выберите конкретный GOAL_ID из выпадающего списка в настройках
3. Нажмите «Сохранить» и перезапустите сервер (`./run.sh`)

## 7. Технические детали

### 7.1 Архитектура

```
Яндекс.Метрика API ──OAuth──▶ backend (Fastify, порт 4000) ──▶ SQLite
                                      │
                           frontend (React+Vite, порт 5173)
                                      │
                           snapshot-builder ──▶ DOCX / PDF
```

### 7.2 Технологический стек

- **Backend:** Node 20, TypeScript, Fastify 4, Zod, better-sqlite3
- **Frontend:** React 18, Vite 5, TailwindCSS, Apache ECharts, TanStack Query
- **Отчёты:** docx, puppeteer-core
- **Тесты:** Vitest, Playwright
- **CI/CD:** GitHub Actions (CI, E2E, Security)

### 7.3 Структура проекта

```
.
├── code/
│   ├── backend/    # Fastify API, Metrika-клиент, SQLite, отчёты
│   ├── frontend/   # React + Vite дашборд
│   └── shared/     # Общие типы и константы
├── docs/           # Документация
├── data/           # SQLite + отчёты (gitignored)
├── setup.sh        # Одна команда: install → init → start
├── init.sh         # Интерактивная инициализация .env
└── run.sh          # Запуск
```
