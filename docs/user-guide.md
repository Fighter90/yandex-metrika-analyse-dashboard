# Руководство пользователя

> 🇷🇺 Русский · [🇬🇧 English](en/user-guide.md)

Как пользоваться дашбордом и генерировать отчёты. Для команды трека «Конверсии и лидген».

> ✅ **Статус: рабочий продукт (v0.10.0).** Все страницы ниже, живой sync Метрики, детерминированные
> DOCX/PDF-отчёты и опциональный AI-анализ доступны. Запуск: `./setup.sh`.

## 1. Что это и зачем

Инструмент отвечает на один вопрос: **откуда берутся платные билеты и как добрать до 300**.
Он не заменяет Яндекс.Метрику, а превращает её сырые цели в понятную картину:
разбивку по каналам, воронку «визит → заявка → оплата», и дисциплину гипотез.

Главный принцип: **заявка ≠ оплата**. Цель в Метрике (заявка) и реальная оплата
разделены везде в интерфейсе.

## 2. Запуск

См. [runbook.md](runbook.md). Коротко: `./setup.sh` (или `pnpm install && ./init.sh && ./run.sh`) →
браузер на `http://localhost:5173`.

## 3. Дашборд: страницы

Глобальные фильтры (шапка): период (7д / 14д / с начала кемпа / произвольный),
каналы, переключатель B2C / B2C+B2B / B2B, показ архивных целей, кнопка «Sync now».

| Страница       | Что показывает                                                                                                                                     |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Overview**   | KPI-стрип (цель 300, заявки, gap), графики: визиты/заявки по дням, микс каналов, **топ стран**, **доля устройств**, weak spots                     |
| **Traffic**    | бар каналов + таблица UTM-разбивки, бейдж низкого покрытия UTM                                                                                     |
| **Audience**   | графики и таблицы по стране и устройству: визиты / пользователи / заявки / CR                                                                      |
| **Behavior**   | страницы входа (startURL) и выхода (exitURL): графики + таблицы (визиты / отказы / заявки / CR)                                                    |
| **Trends**     | линия визитов и заявок по дням + сравнение неделя-к-неделе (WoW) со стрелками                                                                      |
| **Funnel**     | воронка «заявка ≠ оплата»: Визиты → Заявки B2C → Билеты B2B (в работе) → Оплачено B2B, поэтапная конверсия                                         |
| **Report**     | сборка неизменяемого snapshot, опц. AI-анализ, экспорт DOCX/PDF                                                                                    |
| **B2B**        | CRUD-таблица сделок, pipeline по этапам                                                                                                            |
| **Hypotheses** | AI-генерация problem/solution гипотез, ICE-сортировка, риски, планы проверки                                                                       |
| **Decisions**  | Decision Log: карточки DL-{N}, timeline, фильтр по исходу                                                                                          |
| **История**    | Список сформированных снапшотов по датам, период, количество                                                                                       |
| **Настройки**  | OAuth-токен, Client ID/Secret, COUNTER_ID, GOAL_ID, ANTHROPIC_API_KEY; кнопки «Сохранить» и «Очистить данные»                                     |

### Откуда эта цифра?

У каждого числа будет debug-панель «Where does this number come from?» — она показывает
запрос к Метрике и `raw_response_id` в SQLite. Никаких выдуманных цифр.

## 4. Работа с гипотезами

Гипотеза сохраняется только в полном формате Воронковой. UI блокирует сохранение, пока нет:

1. **Формат:** Subject (ЦА) · Action · Solution · Condition («…, если …»).
2. **≥3 скрытых допущения**, покрывающих категории **behavior / market / tech**.
3. **≥2 способа проверки** (synthetic CustDev / live / quantitative / market).
4. **ICE** — три ползунка Impact / Confidence / Ease (1–10), у каждого обязательное
   обоснование в одну строку. Итог `I × C × E` (1–1000) с цветной плашкой приоритета.
5. **Светофор** 🟢/🟡/🔴 — каждый с конкретным порогом и метрикой.
6. **Дедлайн** проверки (в днях).

Кнопка «Generate hypothesis with skill» подскажет, как через
`.claude/skills/hypothesis-check/SKILL.md` структурировать сырую идею в Claude и вставить
поля в форму. Это помощник, а не автоматизация.

## 5. Decision Log

Каждая проверенная гипотеза → запись DL-{N}: что проверяли, что узнали (3–5 пунктов),
цитаты/данные (обязательно), исход по светофору, следующий шаг. При сохранении статус
связанной гипотезы автоматически меняется на исход (green/yellow/red). Запись можно
экспортировать в `data/decisions/DL-{N}.md`.

## 6. Генерация отчётов

На странице **Report**:

1. Выберите период в шапке и нажмите **«Сформировать snapshot»** — соберётся и сохранится
   неизменяемый `ReportSnapshot`, покажется сводка KPI и **полный отчёт прямо на экране**
   (тот же контент, что в DOCX/PDF).
2. (Опц.) **«Сгенерировать AI-анализ»** — вызывает Anthropic из чисел снапшота и добавляет
   помеченную секцию «AI-анализ». Нужен `ANTHROPIC_API_KEY`; без него — понятное сообщение, отчёт
   соберётся без этой секции.
3. **Export DOCX** или **Export PDF** — рендерит файл в `data/reports/{snapshotId}.{docx|pdf}`.

Отчёт **детальный (≥25 страниц A4)**: каждая секция начинается с новой страницы. Секции: Cover,
Executive Summary («заявка ≠ оплата»), Методология, Воронка визит→заявка→оплата, Анализ по каналам,
Приоритизация по ICE, Define (полная карточка каждой проблемной гипотезы), Develop (полная карточка
каждой решенческой гипотезы), Deliver/Decision Log (находки, доказательства, исход, следующий шаг),
AI-анализ (если сгенерирован), топ-разбивки (UTM, гео+устройство, страницы входа/выхода), Дорожная
карта, Глоссарий, Data Appendix. Каждая гипотеза раскрыта полностью: формулировка Воронковой, скрытые
допущения по категориям, методы проверки, разбор ICE с обоснованием каждого балла, светофор-критерии,
дедлайн и статус — с детальным обоснованием по каждому пункту.

Отчёт строится из неизменяемого **snapshot**: один и тот же `snapshotId` даёт один и тот же контент в
DOCX, PDF и на экране (в render-пути нет `Date.now()`/LLM — AI-нарратив генерится один раз и
сохраняется; общий источник контента — `reportSections` из `@pca/shared`). PDF требует локального
Chrome через `PUPPETEER_EXECUTABLE_PATH`; DOCX — без зависимостей.

## 7. Принципы, о которых стоит помнить

- Любая цифра прослеживается до `raw_responses` — если не прослеживается, это баг.
- B2B входит в KPI 300 и ведётся вручную на странице B2B.
- Методология из `.claude/skills/` менять без отметки в
  `docs/methodology-hypotheses.md` нельзя.
