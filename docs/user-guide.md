# Руководство пользователя

> 🇷🇺 Русский · [🇬🇧 English](en/user-guide.md)

Как пользоваться дашбордом и генерировать отчёты. Для команды трека «Конверсии и лидген».

> ✅ **Статус: рабочий продукт (v2.5.7).** Дашборд из 12 страниц: Обзор, Трафик, Поведение, Воронка,
> Цели, Отчёт, История, Настройки, Справка. Глобальные фильтры до 1 года, AI-анализ отчёта,
> GOST-форматированные DOCX/PDF. Запуск: `./setup.sh`.

## 1. Что это и зачем

Инструмент отвечает на один вопрос: **откуда берутся платные билеты и как добрать до 300**.
Он не заменяет Яндекс.Метрику, а превращает её сырые цели в понятную картину:
разбивку по каналам, воронку «визит → заявка → оплата», и дисциплину гипотез.

Главный принцип: **заявка ≠ оплата**. Цель в Метрике (заявка) и реальная оплата
разделены везде в интерфейсе.

## 2. Запуск

См. [runbook.md](runbook.md). Коротко: `./setup.sh` (или `pnpm install && ./init.sh && ./run.sh`) →
браузер на `http://localhost:5173`.

## 3. Дашборд: страницы

В дашборде **12 страниц** (верхнее меню; на экранах < 1024px — гамбургер-меню).

Глобальные фильтры (шапка, на всех страницах кроме «Настройки» и «Справка»): пресеты периода
**7д / 14д / 30д / 90д / 1г**, кастомный date picker (от/до, макс. 365 дней), переключатель сегмента
**B2C / B2C+B2B / B2B**, чекбокс «Архивные цели».

| Страница      | URL         | Что показывает                                                                                                                                        |
| ------------- | ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Обзор**     | `/`         | KPI-стрип (цель 300, заявки, gap), графики визитов/заявок по дням, микс каналов, топ стран, доля устройств, weak spots, цветные бейджи-выводы (🟢/🔴) |
| **Трафик**    | `/traffic`  | бар каналов, визиты vs заявки, таблица каналов с CR, UTM-разбивка, бейдж низкого покрытия UTM, insights                                               |
| **Поведение** | `/behavior` | страницы входа (startURL) и выхода (exitURL): графики + таблицы (визиты / отказы / заявки / CR), рекомендации                                         |
| **Воронка**   | `/funnel`   | воронка «заявка ≠ оплата»: Визиты → Заявки B2C → B2B pipeline → Оплачено B2B, анализ потерь, CR по каналам, B2B по этапам                             |
| **Цели**      | `/goals`    | прогресс-ринг к 300 билетам, метрики, B2B-сделки, рекомендации на основе данных                                                                       |
| **Отчёт**     | `/report`   | сборка snapshot, AI-анализ (5 секций, HTML), отчёт прямо на экране, экспорт DOCX/PDF, перестроение                                                    |
| **История**   | `/history`  | список сохранённых снапшотов по датам; «Просмотреть» открывает снапшот без перегенерации                                                              |
| **Настройки** | `/settings` | OAuth-токен, Client ID/Secret, COUNTER_ID, GOAL_ID (select из Метрики), ANTHROPIC_API_KEY; «🔄 Обновить данные» с прогресс-баром                      |
| **Справка**   | `/help`     | встроенная документация: описание всех страниц, фильтров, FAQ, глоссарий                                                                              |

> Страницы «Настройки» и «Справка» — без панели фильтров.

### Откуда эта цифра?

Любое число в дашборде и отчёте прослеживается до сырого ответа Метрики в `raw_responses` (SQLite).
Никаких выдуманных цифр — это инвариант anti-hallucination.

## 4. Работа с гипотезами

Гипотеза сохраняется только в полном структурированном формате. UI блокирует сохранение, пока нет:

1. **Формат:** Subject (ЦА) · Action · Solution · Condition («…, если …»).
2. **≥3 скрытых допущения**, покрывающих категории **behavior / market / tech**.
3. **≥2 способа проверки** (synthetic CustDev / live / quantitative / market).
4. **ICE** — три ползунка Impact / Confidence / Ease (1–10), у каждого обязательное
   обоснование в одну строку. Итог `I × C × E` (1–1000) с цветной плашкой приоритета.
5. **Светофор** 🟢/🟡/🔴 — каждый с конкретным порогом и метрикой.
6. **Дедлайн** проверки (в днях).

Кнопка «Generate hypothesis with skill» подскажет, как через
`.claude/skills/hypothesis-check/SKILL.md` структурировать сырую идею в Claude и вставить
поля в форму. Это помощник, а не автоматизация.

## 5. Decision Log

Каждая проверенная гипотеза → запись DL-{N}: что проверяли, что узнали (3–5 пунктов),
цитаты/данные (обязательно), исход по светофору, следующий шаг. При сохранении статус
связанной гипотезы автоматически меняется на исход (green/yellow/red). Запись можно
экспортировать в `data/decisions/DL-{N}.md`.

## 6. Генерация отчётов

На странице **Report**:

1. Выберите период в шапке и нажмите **«Сформировать snapshot»** — соберётся и сохранится
   неизменяемый `ReportSnapshot`, покажется сводка KPI и **полный отчёт прямо на экране**
   (тот же контент, что в DOCX/PDF).
2. (Опц.) **«Сгенерировать AI-анализ»** — вызывает Anthropic из чисел снапшота и добавляет
   помеченную секцию «AI-анализ». Нужен `ANTHROPIC_API_KEY`; без него — понятное сообщение, отчёт
   соберётся без этой секции.
3. **Export DOCX** или **Export PDF** — рендерит файл в `data/reports/{snapshotId}.{docx|pdf}`.

Отчёт **детальный**, в GOST-форматировании (Times New Roman 14pt, титульная страница, оглавление,
нумерованные разделы, номера страниц; поддерживается markdown — таблицы, жирный/курсив, списки).
Секции: Cover, Executive Summary («заявка ≠ оплата»), Методология, Воронка визит→заявка→оплата,
Анализ по каналам, Приоритизация по ICE, Define (полная карточка каждой проблемной гипотезы),
Develop (полная карточка каждой решенческой гипотезы), Deliver/Decision Log (находки, доказательства,
исход, следующий шаг), AI-анализ (если сгенерирован), топ-разбивки (UTM, гео+устройство, страницы
входа/выхода), Дорожная карта, Глоссарий, Data Appendix. Каждая гипотеза раскрыта полностью:
формулировка по структурированному формату, скрытые допущения по категориям, методы проверки, разбор
ICE с обоснованием каждого балла, светофор-критерии, дедлайн и статус.

Отчёт строится из неизменяемого **snapshot**: один и тот же `snapshotId` даёт один и тот же контент в
DOCX, PDF и на экране (в render-пути нет `Date.now()`/LLM — AI-нарратив генерится один раз и
сохраняется; общий источник контента — `reportSections` из `@pca/shared`). PDF требует локального
Chrome через `PUPPETEER_EXECUTABLE_PATH`; DOCX — без зависимостей.

## 7. Принципы, о которых стоит помнить

- Любая цифра прослеживается до `raw_responses` — если не прослеживается, это баг.
- B2B входит в KPI 300 и ведётся вручную (таблица `b2b_manual`, CRUD через `POST /api/b2b`;
  отдельная UI-страница B2B запланирована, в текущем меню её нет).
- Методология из `.claude/skills/` менять без отметки в
  `docs/methodology-hypotheses.md` нельзя.
