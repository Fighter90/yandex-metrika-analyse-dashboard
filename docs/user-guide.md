# Руководство пользователя

> 🇷🇺 Русский · [🇬🇧 English](en/user-guide.md)

Полное руководство по дашборду и отчётам ProductCamp Conversion Analytics Tool — для команды трека
«Конверсии и лидген» (продакты, маркетологи, инженеры-волонтёры).

> ✅ **Статус: рабочий продукт (v2.8.3).** Дашборд из 9 страниц (Обзор, Трафик, Поведение, Воронка,
> Цели, Отчёт, История, Настройки, Справка). Глобальные фильтры до 1 года (на мобиле — bottom-sheet),
> AI-анализ отчёта в 6 секций (включая «Итоговый вывод»), AI-гипотезы и AI Decision Log внутри отчёта,
> DOCX/PDF по ГОСТ Р 7.32-2017 с титульной страницей, «Содержанием» и **графиками-изображениями**
> (бар каналов, воронка, микс) + блок 🟢/🔴 у каждого графика. B2B-пайплайн (kanban + drawer) в
> Настройках. Доступность WCAG AA. Запуск: `./setup.sh`.

Смежные документы: [быстрый старт / развёртывание](quickstart.md) · [runbook](runbook.md) ·
[архитектура](architecture.md) · [модель данных](data-model.md).

## 1. Что это и зачем

Инструмент отвечает на один вопрос: **откуда берутся платные билеты и как добрать до 300**.
Он не заменяет Яндекс.Метрику, а превращает её сырые цели в понятную картину: разбивку по каналам,
воронку «визит → заявка → оплата», приоритизированные гипотезы роста и Decision Log.

Два сквозных принципа:

- **Заявка ≠ оплата.** Достижение цели в Метрике (заявка) и фактическая оплата разделены везде в
  интерфейсе и в отчёте. Прогресс к KPI считается по оплатам.
- **Anti-hallucination.** Любое число на дашборде и в отчёте прослеживается до сырого ответа Метрики
  в таблице `raw_responses` (SQLite). Выдуманных цифр нет — это инвариант.

## 2. Запуск (коротко)

Подробно — в [quickstart.md](quickstart.md). Самый короткий путь:

```bash
git clone git@github.com:Fighter90/yandex-metrika-analyse-dashboard.git
cd yandex-metrika-analyse-dashboard
./setup.sh          # install → init → start
```

Откроется `http://localhost:5173` (API — `http://localhost:4000`, Swagger — `/docs`).
Без OAuth-токена дашборд поднимется на демо-данных (`pnpm seed`).

### 3.1 Обзор (`/`)

- **KPI-стрип:** цель (300), достижения основной цели (метка «Оплат» при цели-покупке, иначе
  «Заявок B2C» с пометкой «заявка ≠ оплата»), gap до цели.
- **Дайджест за неделю:** визиты и заявки за последние 7 дней с WoW-дельтой (сравнение с предыдущими
  7 днями), топ-канал по визитам, главное слабое место. Источник — `channel_stats`.
- **Графики:** визиты/заявки по дням, микс каналов, топ стран, доля устройств.
- **Слабые места:** каналы с трафиком, но конверсией ниже средней — кандидаты на гипотезы.
- **Подписи под графиками:** блок 🟢 Корректно / 🔴 Внимание / 💡 Рекомендация (см. §3.10).

### 3.2 Трафик (`/traffic`)

- Бар-чарт визитов по каналам; график «визиты vs заявки» (какой трафик реально конвертит).
- **UTM-Sankey** «источник → кампания → заявки» — поток атрибуции. Кнопка **«⛶ Развернуть на весь
  экран»** открывает читаемую полноэкранную версию (в инлайн-виде поток может быть мелким).
- Таблица каналов (визиты / пользователи / заявки / CR) и UTM-разбивка (source / medium / campaign).
- UTM-покрытие = визиты с UTM-метками / все визиты; при < 70% выводится предупреждение.

### 3.3 Поведение (`/behavior`)

- Страницы входа (`ym:s:startURL`) и выхода (`ym:s:exitURL`): графики bounce/CR + таблицы.
- ⚠️ **Страницы выхода** на живых данных счётчика ProductCamp пусты — `ym:s:exitURL` не поддерживается
  Stat API. Блок скрывается при отсутствии данных (это ограничение API, не баг).

### 3.4 Воронка (`/funnel`)

- 4 этапа: Визиты → Заявки B2C → B2B pipeline → Оплачено B2B (этапы не суммируются — разные метрики).
- Анализ потерь, воронка по каналам, CR по каналам, разбивка B2B по этапам, проблемные страницы.

### 3.5 Цели (`/goals`)

- **Прогресс-ринг к 300 оплаченным билетам.** Прогресс считает подтверждённые оплаты:
  **B2B paid + достижения цели-покупки из Метрики** (когда основная цель — покупка / `e_purchase`,
  её reaches = оплаты). При цели-заявке учитывается только B2B paid (заявка ≠ оплата).
- Метрики: визиты, общий CR, число B2B-сделок, процент прогресса.
- Таблица B2B-сделок; «Что если»-симулятор (прикидка эффекта роста трафика и CR); рекомендации.

### 3.6 Отчёт (`/report`)

См. §6 — сборка снапшота, AI-анализ, экспорт DOCX/PDF.

### 3.7 История (`/history`)

Список сохранённых снапшотов (ID, дата, период). «Просмотреть» открывает сохранённый отчёт
(с AI-нарративом, если он был сгенерирован) без повторной генерации.

### 3.8 Настройки (`/settings`)

- **Текущий счётчик** (COUNTER_ID) — из `.env` или health API.
- **OAuth-токен**, **Client ID/Secret** — для авторизации в Метрике.
- **GOAL_ID** — combobox с поиском и группами «Авто / Активные / Архивные»: `0` = авто-определение
  основной цели оплаты/покупки, иначе фиксированная цель.
- **ANTHROPIC_API_KEY** — ключ для AI-анализа. Если ключ уже задан, под полем показывается его маска
  («Текущий ключ: sk-a\*\*\*\*AA»); оставьте поле пустым, чтобы не менять ключ (v2.9.1).
- **🔄 Обновить данные из Метрики** — синхронизация с прогресс-баром (10 стадий: подключение → цели →
  каналы → UTM → гео → страницы входа → страницы выхода → запись в БД → перегенерация → готово).
- **Секция «B2B-пайплайн»** — kanban из 4 этапов + форма добавления; кнопка «Развернуть» открывает
  полную таблицу в drawer (см. §5).

**Сохранение настроек.** Кнопка «Сохранить» пишет значения в `.env`, **сохраняя остальные строки и
комментарии** (бэкенд обновляет только переданные ключи). Маскированные/пустые секреты НЕ
перезаписываются; `COUNTER_ID` отправляется только если изменён. После сохранения перезапустите
сервер (`./run.sh`), чтобы настройки применились.

Секреты маскируются (`****`) и не отправляются обратно на сервер при сохранении; `COUNTER_ID`
передаётся только если пользователь его явно изменил.

### 3.9 Справка (`/help`)

Встроенная документация: описание каждой страницы и графиков, фильтры, FAQ, глоссарий
(CR, bounce rate, UTM, B2C/B2B, ICE и др.).

### 3.10 Подписи под графиками

Под графиками/таблицами выводится блок из трёх строк, расширяющий anti-hallucination:

- **🟢 Корректно** — источник данных в SQLite и что он означает.
- **🔴 Внимание** — известное ограничение (например, `exitURL` не отдаётся Метрикой).
- **💡 Рекомендация** — что проверить или как использовать данные.

## 4. Фильтры (шапка дашборда)

Панель фильтров вверху всех страниц (кроме «Настройки» и «Справка») влияет на все графики и таблицы:

- **Пресеты периода:** 7д / 14д / 30д / 90д / 1г.
- **📅 Даты** — кастомный период (от → до → «Применить»), максимум **365 дней**; при превышении или
  если дата окончания раньше начала — выводится ошибка.
- **Сегмент:** B2C (потребительские каналы) / B2C+B2B (все, по умолчанию) / B2B (только B2B-каналы).
- **Архивные цели** — показать/скрыть удалённые/заархивированные цели Метрики.
- **На мобиле (< 768px):** фильтры сворачиваются в кнопку **«Фильтры»** → выезжающий bottom-sheet
  (бэкдроп + «Готово»).

## 5. B2B-пайплайн (в Настройках)

Ручной ввод B2B-сделок — в сворачиваемой секции **«B2B-пайплайн»** на странице **Настройки**:

- Компактный **kanban из 4 этапов**: lead → negotiation → invoiced → paid.
- Форма добавления (компания / билеты / этап); кнопка «Развернуть» открывает полную редактируемую
  таблицу в drawer (удаление сделок и т.д.).
- Оплаченные билеты (`paid`) засчитываются в KPI 300 и учитываются на страницах «Воронка» и «Цели».
- Данные хранятся в таблице `b2b_manual`; CRUD через `POST/PATCH/DELETE /api/b2b`.

Прямой URL `/b2b` перенаправляет на `/settings`.

## 6. Генерация отчётов (DOCX/PDF + AI)

На странице **Отчёт** (`/report`):

1. Выберите период в шапке и нажмите **«Сформировать срез данных»** — соберётся и сохранится
   неизменяемый `ReportSnapshot`; покажется сводка KPI и **полный отчёт прямо на экране** (тот же
   контент, что в DOCX/PDF), включая AI-гипотезы и AI Decision Log.
2. (Опционально) **«Сгенерировать AI-анализ»** — вызывает Anthropic из чисел снапшота и добавляет
   AI-нарратив в 6 секций. Нужен `ANTHROPIC_API_KEY`; без него выводится понятное сообщение, а отчёт
   собирается без этой секции.
3. **Export DOCX** / **Export PDF** — скачивает файл (поток байтов с `Content-Disposition: attachment`).
4. **Перестроить отчёт** — пересобирает снапшот за текущий период.

### 6.1 Структура отчёта (ГОСТ Р 7.32-2017)

- **Титульная страница:** организация (ProductCamp), трек, название работы, период, идентификатор
  среза данных, год.
- **Содержание** (нумерованные разделы) отдельной страницей.
- Разделы: Краткие итоги (заявка ≠ оплата), Рекомендации 🟢/🔴, Методология, Воронка, B2B-пайплайн,
  **«Каналы: трафик и конверсия» + график-бар**, **«Воронка визит→заявка→оплата» + график-воронка**,
  **«Каналы: структура трафика» + график-микс** (у каждого графика — блок 🟢/🔴), Приоритизация
  гипотез, **AI-гипотезы** (из `snapshot.generatedHypotheses`), **AI Decision Log** (из
  `snapshot.generatedDecisions`), AI-анализ (если сгенерирован), топ-разбивки (UTM / гео+устройство /
  страницы входа/выхода), глоссарий, приложение с данными.
- **Две секции по каналам имеют различимые названия** («трафик и конверсия» — детальная таблица +
  бар; «структура трафика» — микс) — без дублирующихся заголовков (v2.9.0).
- **Приложение с данными — компактное**: агрегированные итоги по каналам за период (не построчный
  посуточный дамп). Полная посуточная разбивка восстановима из `raw_responses` (v2.9.0).
- **Пустые разделы не печатаются (v2.9.3):** раздел только с заголовком и заглушкой («Нет данных…»,
  «Завершённых проверок пока нет», «Гипотезы не заведены…») отбрасывается — в отчёте остаются только
  разделы с реальными данными. На живых данных так уходит «Топ страниц выхода» (exitURL пуст).
- Оформление: Times New Roman 14pt, межстрочный 1.5, поля 30/15/20/20 мм, нумерация страниц,
  выравнивание абзацев по ширине, без пустых блоков-заполнителей.
- **Согласованность KPI (v2.9.3):** Gap и число оплат на «Обзоре», «Целях», в превью и DOCX/PDF
  совпадают; при цели-покупке (e_purchase) её достижения учитываются как оплаты (заявка=оплата для
  такой цели), Gap = 300 − оплаты.

### 6.2 AI-анализ — 6 секций

Итог · Каналы, UTM и Аудитория · Страницы и Воронка · Риски и Рекомендации · Гипотезы и Дорожная карта ·
**Итоговый вывод** (подробный управленческий синтез: вердикт по цели + следующие шаги). Нарратив
рендерится полным HTML (заголовки, списки, таблицы, цитаты) — без «сырых» markdown-тегов. В DOCX/PDF
из строк AI дополнительно срезаются HTML-теги: `<strong>/<b>`→`**`, `<em>/<i>`→`*`, прочие
(`<p>`,`<br>`,…) удаляются, чтобы в документе не было «протекающего» HTML (v2.9.3).

AI-нарратив **очищается** при сборке отчёта: срезаются маркеры markdown-заголовков (`#`…`######`),
цитат (`>`), **ведущие эмодзи** и **собственная нумерация** AI-заголовков («1.», «Шаг 1.», «Раздел 1.»)
— чтобы в DOCX/PDF не «протекали» `##`/`####`/`>`, не было заголовков с эмодзи и двойной нумерации
вида «11. 1. …» (v2.9.1). Эмодзи внутри абзаца сохраняются как акцент; маркированные списки («- …»)
рендерятся как настоящие списки. Каждая AI-секция ограничена по объёму (≤ 35 строк, с пометкой о
сокращении — полный текст в `snapshot.aiNarrative`), чтобы не раздувать отчёт. Сам AI-промпт
(`AI_FORMAT_RULES`) запрещает markdown-заголовки, свою нумерацию, эмодзи в заголовках и HTML. Кнопка
«Сгенерировать AI-анализ» блокируется после успешной генерации (нельзя сгенерировать дважды для среза).

### 6.3 Детерминизм

Отчёт строится из неизменяемого снапшота: один `snapshotId` → один и тот же контент в DOCX, PDF и на
экране. В render-пути нет `Date.now()` и нет LLM-вызовов — AI-нарратив и графики-PNG генерируются один
раз и сохраняются; общий источник контента — `reportSections` из `@pca/shared`. Графики рендерятся
server-side (headless ECharts) и встраиваются идентично в DOCX (`ImageRun`) и PDF (`<img data:>`).
PDF требует локального Chrome (авто-поиск или `PUPPETEER_EXECUTABLE_PATH`); DOCX — без зависимостей.

## 7. Источники данных

### 7.1 Яндекс.Метрика (Stat API)

| Запрос                         | Что приносит                                        |
| ------------------------------ | --------------------------------------------------- |
| `traffic-by-source`            | визиты, пользователи, отказы, заявки, CR по каналам |
| `utm-breakdown`                | разбивка по UTM: source / medium / campaign         |
| `geo-device-breakdown`         | страна + тип устройства                             |
| `page-behavior` (startURL)     | страницы входа: визиты, отказы, заявки              |
| `exit-page-behavior` (exitURL) | страницы выхода (на живых данных часто пусто)       |

Все запросы используют `attribution=lastsign` (паритет с интерфейсом Метрики) и трафик по одному
измерению `ym:s:lastTrafficSource`. Проценты Метрики (bounce/CR) нормализуются в долю 0–1.

### 7.2 Ручной ввод (B2B)

`b2b_manual`: company, tickets, stage (lead/negotiation/invoiced/paid), amountRub, contactEmail,
notes, dateAdded, datePaid. Полная модель — в [data-model.md](data-model.md).

## 8. Устранение проблем

- **Дашборд не запускается:** `node --version` ≥ 20, `pnpm --version` ≥ 9; `rm -rf node_modules && pnpm install`.
- **Нет данных из Метрики:** проверьте `COUNTER_ID` (> 0) и `YANDEX_OAUTH_TOKEN` в `.env`; нажмите
  «Обновить данные» в Настройках; смотрите логи backend (`pnpm --filter @pca/backend dev`).
- **PDF не экспортируется** (`executablePath … must be specified`): задайте в `.env`
  `PUPPETEER_EXECUTABLE_PATH=/Applications/Google Chrome.app/Contents/MacOS/Google Chrome` или
  установите Chrome.
- **AI-анализ не генерируется:** проверьте `ANTHROPIC_API_KEY`, интернет; повторите через ~30с.
- **GOAL_ID не определяется автоматически:** убедитесь, что в Метрике есть цель типа «покупка»/«оплата»,
  или выберите цель вручную в combobox Настроек.
- **Смена цели/исправление единиц:** требует чистой БД (`rm data/productcamp.sqlite*`), т.к. upsert
  накапливает строки и не удаляет устаревшие.

## 9. CLI-команды

| Команда                                         | Описание                                                   |
| ----------------------------------------------- | ---------------------------------------------------------- |
| `./setup.sh`                                    | всё одной командой: install → init → start                 |

## 10. Принципы, о которых стоит помнить

- Любая цифра прослеживается до `raw_responses` — если не прослеживается, это баг.
- Заявка ≠ оплата: эти метрики не суммируются; прогресс к KPI — по оплатам.
- B2B входит в KPI 300 и ведётся вручную через секцию «B2B-пайплайн» в Настройках.
- Методологию из `.claude/skills/` нельзя менять без отметки в `docs/methodology-hypotheses.md`.
- Новые зависимости вне зафиксированного стека — только через ADR в `docs/decisions/`.
