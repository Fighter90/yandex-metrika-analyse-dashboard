# ADR-005 — ICE как произведение (I × C × E), а не среднее

- **Статус:** принято
- **Дата:** 2026-05-22
- **Контекст:** для приоритизации гипотез по методологии Воронковой нужен числовой балл ICE из
  трёх параметров — Impact, Confidence, Ease (каждый 1–10). Распространён вариант «среднее»
  ((I+C+E)/3), но он маскирует однобокие гипотезы.

## Решение

- **ICE = I × C × E** (произведение), диапазон **1–1000**. Реализация: `ICE_CONFIG` и
  `iceScore`/`iceBucket` в `code/shared/src/` (общие для backend и frontend, чтобы балл считался
  одинаково везде).
- Каждый параметр требует **обоснования** (rationale) — балл без обоснования не сохраняется.
- Бакеты приоритета (`iceBucket`) делят 1–1000 на зоны для бейджей в UI.

## Обоснование

- Произведение **резко поднимает** гипотезы с высокими значениями по всем трём осям и **наказывает
  однобокие**: «Impact=10, Confidence=2, Ease=2» = **40**, а не **≈4.7** в среднем — разница в
  приоритете становится наглядной.
- Единый источник истины (`@pca/shared`) исключает расхождение между дашбордом и отчётом.
- 1–1000 — интуитивная шкала для сортировки и порогов светофора.

## Последствия

- Балл нелинеен: рост на 1 по слабой оси даёт больший прирост, чем по сильной — это намеренно
  стимулирует «добивать» слабые места гипотезы.
- Пороги бакетов подобраны под шкалу 1–1000; при изменении формулы их нужно пересматривать.

## Альтернативы

- **Среднее (I+C+E)/3** — отклонено: сглаживает риски, однобокие гипотезы выглядят не хуже
  сбалансированных.
- **Взвешенная сумма** — отклонено: добавляет субъективные веса без явного выигрыша против
  произведения для данной задачи.
