# Specs — Spec-Driven Development (SDD)

> 🇷🇺 Русский · спецификации фич этого проекта. Лёгкий процесс, без внешних плагинов — всё в репозитории.

Здесь живут **спецификации функционала**: каждая нетривиальная фича сначала описывается спекой,
ревьюится, и только потом реализуется. Это закрывает разрыв «идея → код» и держит решения
прослеживаемыми (в духе anti-hallucination всего проекта).

## Когда писать спеку

- **Обязательно:** фича затрагивает > 1 файла или > ~30 минут работы; меняет модель данных, API,
  методологию или KPI-математику.
- **Не нужно:** опечатки, разовые правки, чистка, обновление зависимостей, документация.

Это согласуется с правилом репозитория: новый функционал — через ветку под итерацию, с тестами и
100% покрытием (см. [../testing-strategy.md](../testing-strategy.md)).

## Процесс: spec → plan → tests → impl

1. **Spec** — скопируйте [TEMPLATE.md](TEMPLATE.md) в `NNN-kebab-title.md` (следующий свободный номер),
   заполните разделы. Зафиксируйте _что_ и _почему_, измеримые критерии приёмки, влияние на данные/API/UI.
2. **Review** — спека ревьюится (PR только со спекой допустим). Статус в шапке: `draft → accepted`.
   Спека с открытыми вопросами не принимается.
3. **Plan** — раздел «План реализации» спеки: список файлов, миграции, порядок задач (TDD-шаги).
4. **Tests first** — пишутся падающие тесты по критериям приёмки (red), затем минимальная реализация
   (green), затем рефактор. Никогда не реализуем happy-path без предварительного теста.
5. **Impl** — реализация по плану; каждый PR ссылается на спеку (`docs/specs/NNN-…`). По завершении
   статус спеки → `done`, добавляется ссылка на PR/релиз.

## Жизненный цикл и статусы

`draft` (черновик) → `accepted` (принята к реализации) → `done` (реализовано, со ссылкой на PR) ·
`superseded` (заменена другой спекой — указать какой).

## Соглашения

- Имя файла: `NNN-kebab-title.md`, `NNN` — трёхзначный сквозной номер.
- Один файл = одна фича (как один PR = одно логическое изменение).
- Спека — единый источник _намерения_; реализация может уточнять детали, но расхождения с принятой
  спекой фиксируются обновлением спеки, а не молча.
- Числа и метрики в спеке, опирающиеся на данные, прослеживаются до Метрики/`raw_responses`.
