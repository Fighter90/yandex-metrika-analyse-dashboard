## Продукт

- **Название:** ProductCamp Conversion Analytics Tool
- **Что делает (в одном предложении):** локальный аналитический инструмент, который вытаскивает данные Яндекс.Метрики счётчика ProductCamp, строит интерактивный дашборд по конверсиям и генерирует отчёт с проверяемыми гипотезами по росту платных продаж.
- **Стадия:** MVP (≈2 недели до старта кемпа).
- **Сайт / репозиторий:** локальный инструмент, запуск через `./run.sh`.

## Как это работает (локально, одной командой `./run.sh`)

Инструмент разворачивается **только на локальном компьютере** — никакого облачного деплоя.
`./run.sh` поднимает весь цикл:

1. **Parse** — по OAuth тянет данные Яндекс.Метрики (счётчик `54280963`) и кэширует
   сырые ответы в SQLite (`raw_responses`) для offline и воспроизводимости.
2. **Dashboard** — локальный дашборд (frontend `:5173`, API `:4000`): разбивка по каналам,
   воронка визит → заявка → оплата, weak spots, B2B. Каждая цифра прослеживается до `raw_responses`.
3. **Hypotheses** — на данных дашборда формулируются гипотезы по методологии Воронковой
   (формат + ≥3 допущения + ≥2 метода + ICE=product + светофор + дедлайн); решения
   фиксируются в Decision Log.
4. **Report** — из неизменяемого `ReportSnapshot` строятся **DOCX и PDF** с гипотезами,
   выведенными из данных дашборда и Метрики. Один `snapshotId` → байт-идентичные файлы.
   В render-пути нет LLM-вызовов и `Date.now()`.

## ЦА

- **Кто целевой пользователь:** команда трека «Конверсии и лидген» ProductCamp — Лиза (лидирует направление), Сергей (аналитик/инженер), волонтёры команды маркетинга.
- **Какая у него роль / должность:** продакты, маркетологи, инженеры в роли волонтёров-аналитиков.
- **В какой компании / контексте он работает:** команда ProductCamp, дедлайн < 3 недель, ограниченное время, фокус на 1–2–3 самых важных задачах с быстрым эффектом.

## Боль

- **Основная боль:** разрыв между формальными целями в Метрике (>300 заявок) и фактическими оплатами (~150). Не понятно, какой трафик приносит реальные оплаты, а какой — пустые заявки.
- **Что делается сейчас:** ручные выгрузки из Метрики в Excel, отсутствие единого дашборда, гипотезы формулируются в чатах без приоритизации и валидации данными.
- **Почему неудобно:** медленно, нет воспроизводимости, нет связи «гипотеза → данные → решение», волонтёры не могут включиться без 30-минутного брифа.

## Гипотеза, которую сейчас проверяем (для самого инструмента)

> **Формат Воронковой:** «{ЦА} {готова сделать / готова заплатить за / переключится на} {решение}, если {условие}».

**Текущая гипотеза:**
Команда трека «Конверсии и лидген» **готова перейти с ручных выгрузок Excel на этот инструмент**, если он за один `./run.sh` показывает разбивку платных билетов по каналам с UTM и позволяет привязать к каждой проблеме конкретные данные Метрики.

**Дата начала проверки:** старт работы над инструментом.
**Дата дедлайна решения:** за неделю до старта ProductCamp.

## Метрики успеха проверки

- **🟢 Зелёный свет (продолжаем):** ≥3 человека из команды трека пользуются инструментом самостоятельно, ≥5 problem + ≥5 solution гипотез внесены в систему с полным форматом, ≥3 решения зафиксированы в Decision Log с привязкой к данным.
- **🟡 Жёлтый (pivot):** инструмент работает, но команда продолжает дублировать выгрузки в Excel → упрощаем интерфейс, режем фичи.
- **🔴 Красный (стоп):** через неделю никто, кроме автора, не открыл; команда говорит «проще руками».

## Стек

- **Frontend:** React 18 + Vite + TailwindCSS + shadcn/ui + Apache ECharts + TanStack Table
- **Backend:** Node.js 20 + Fastify + TypeScript strict
- **БД:** SQLite (`better-sqlite3`)
- **AI / LLM:** не используется в production-пути генерации отчёта; на стороне разработки — Claude Code с инструкциями из этого файла и `.claude/skills/`
- **Аналитика:** Яндекс.Метрика Reporting API (OAuth), счётчик 54280963

## Принципы

- **Anti-hallucination above all:** ни одной цифры в отчёте, которая не прослеживается до `raw_responses` в SQLite.
- **Один источник правды для отчётов:** PDF и DOCX рендерятся из одного immutable `ReportSnapshot`. Никаких расхождений.
- **Гипотеза без формата Воронковой — не гипотеза.** UI блокирует сохранение без 3 допущений, 2 методов проверки, светофор-критериев и дедлайна.
- **ICE = I × C × E (product, не среднее).** Сознательное решение, обоснование в ADR `docs/decisions/005-ice-product-vs-mean.md`.
- **Decision Log замыкает цикл.** Решение без записи в `decisions` считается не принятым.
- **Воспроизводимость > красота.** Лучше скромный, но детерминированный отчёт, чем «иногда красивее».
- **Заявка ≠ оплата.** Везде в коде и UI явное разделение.
- **B2B — ручной ввод.** Метрика не покрывает B2B-пайплайн, для этого `b2b_manual` с CRUD.

## Парсер и история данных

Перед любым анализом инструмент **выкачивает максимум доступных агрегатов** из
Яндекс.Метрики и кладёт их в локальную **SQLite** (без Docker — см. ADR-002):

- **Парсер (ETL):** `pnpm sync` тянет полный набор отчётов Stat API (§5.4) — трафик по
  источникам, UTM, воронка, поведение, формы, гео/девайс, временные ряды — по всем целям
  (с учётом порога архивации), дневными чанками.
- **Сырьё → производные:** каждый ответ сохраняется в `raw_responses` (для прослеживаемости),
  затем нормализуется в `channel_stats` и др. Дашборд и отчёты строятся **только** из БД,
  не из живых вызовов API.
- **История по дням:** ключи `raw_responses(query_hash, date_from, date_to)` и
  `channel_stats(date, channel, …)` хранят данные по датам — повторный `sync` дополняет
  историю, не затирая прошлые дни. Это даёт WoW-сравнения и воспроизводимость отчётов.
- **Глубина:** агрегаты Stat API (Logs API/сырые хиты — вне scope, см. ADR при необходимости).

- **Тесты — пирамида, цель 100%:** unit (логика/чистые функции) → functional/integration

  (критерии §14). Каждая итерация поставляется с тестами; entry/bootstrap-файлы исключены
  из coverage, на остальном — порог 100%. См. ADR-006.
- **Версионирование:** SemVer + `CHANGELOG.md` + conventional commits; релиз по тегу
  `v*.*.*` (`release.yml`).
- **Ревью:** AI code-review в CI на каждый PR (`.github/workflows/review.yml`).

## Как работать в этом репозитории (для AI-агентов)

- **Методология обязательна.** Гипотеза — строго формат Воронковой
  (`docs/methodology-hypothesis-voronik.md`), ICE = произведение (`docs/methodology-ice.md`),
  цикл Double Diamond (`docs/methodology-double-diamond.md`). Менять методологию из
  `.claude/skills/` — только с записью в `methodology-hypothesis-voronik.md` и reasoning.
- **Anti-hallucination** (`docs/anti-hallucination.md`): любое число → `raw_responses`; никакого
  `Date.now()`/LLM в render-пути отчёта; заявка ≠ оплата; B2B входит в KPI 300.
- **100% покрытие** (`docs/testing-strategy.md`): каждая фича — с тестами; CI гоняет
  `pnpm coverage` с порогом 100%.
- **Поток работы:** ветка под итерацию → Conventional Commits → локальный гейт
  (`lint` / `format:check` / `typecheck` / `coverage` / `build`) → merge в `main` →
  **следить за пайплайнами** (`gh run list`) и чинить красное.
- **Зависимости вне зафиксированного стека — только через ADR** в `docs/decisions/`.
- **Запуск:** `docs/runbook.md`; дашборд и отчёты: `docs/user-guide.md`; API Метрики:
  `docs/metrika-api-cheatsheet.md`.
