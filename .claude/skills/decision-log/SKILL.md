# Skill — Decision Log

> Принцип: «Если решение не записано — его не было».

---

## Назначение

Превращает каждый цикл проверки гипотезы в запись, которую агент «помнит» при следующем обращении. И вы тоже помните — что для команды важнее.

## Где хранить

Один из вариантов (выбрать один и не смешивать):
- `decision-log.md` рядом с CLAUDE.md
- Раздел в Notion / Obsidian
- Issue в GitHub с лейблом `decision`

## Шаблон записи

```markdown
## DL-{номер} — {короткая суть решения} ({дата})

**Гипотеза:**
> {точная формулировка}

**Что проверяли:**
- Метод: {синтетический CustDev / интервью / данные / рынок}
- Объём: {сколько респондентов / запросов / источников}
- Период проверки: {N дней}

**Что узнали (3-5 пунктов):**
1.
2.
3.

**Цитаты / данные (доказательная база):**
> «…»
> Источник: {имя / ссылка}

**Решение:**
- [ ] Зелёный — продолжаем
- [ ] Жёлтый — pivot: {что меняем}
- [ ] Красный — стоп: {почему окончательно}

**Что делаем дальше:**
- Следующий шаг: {конкретное действие}
- Ответственный:
- Дедлайн:

**Связанные:**
- Предыдущее решение: DL-{N}
- Гипотеза, которая родилась отсюда:

**Авторство:**
- Кто принял решение:
- Кто участвовал в обсуждении:
```

## Промпт для генерации записи

```
На основе нашей сессии проверки гипотезы — заполни Decision Log по шаблону:
{вставьте шаблон выше}

Учти только то, что мы реально проверили в этой сессии. Не выдумывай данные. Если чего-то не было — оставь поле пустым с пометкой "не проверяли".
```

## Что делает Decision Log сильным

- **Цитаты и цифры** — без них это «мне показалось»
- **Связь с предыдущими** — видно, как эволюционирует продукт
- **Авторство** — понятно, кто отвечает
- **Следующий шаг** — иначе решение зависает

## Антипаттерны

- «Мы решили продолжать» без критериев — это не решение, а пауза
- Список «выводов» без действий — мёртвый документ
- Один человек заполняет за всю команду — теряется контекст
- Decision Log без срока пересмотра — устаревает

## Бонус — петля

После заполнения положите ссылку на эту запись в `CLAUDE.md`. Следующий раз, когда вы откроете Claude и спросите «какую гипотезу проверять?», агент учтёт прошлый decision log и не предложит то, что вы уже отбросили.

## Пример DL — ProductCamp «Конверсии и лидген»

```markdown
## DL-001 — Подкаст-трафик хуже доходит до оплаты (2025-XX-XX)

**Гипотеза:**
> Посетитель с подкаст-трафика готов оставить заявку, но не оплачивает в тот же день,
> если оплата требует ручного подтверждения реквизитов.

**Что проверяли:**
- Метод: quantitative (данные Метрики) + synthetic CustDev
- Объём: анализ среза channel_stats за 14 дней + 1 synthetic-сессия
- Период проверки: 5 дней

**Что узнали (3-5 пунктов):**
1. conversion-to-paid подкаста ниже прямого трафика (срез из raw_responses #N).
2. Заявок с подкаста много, оплат — заметно меньше (заявка ≠ оплата).
3. Synthetic-роль CTO бросает корзину при оплате по реквизитам.

**Цитаты / данные (доказательная база):**
> «Если надо счёт и ждать — отложу на потом и забуду»
> Источник: synthetic CustDev, роль «CTO стартапа»
> Данные: raw_response_id=N, slice: channel=podcast vs direct

**Решение:**
- [x] Жёлтый — pivot: проверить solution про мгновенную онлайн-оплату

**Что делаем дальше:**
- Следующий шаг: завести solution-гипотезу (кнопка онлайн-оплаты на новой посадочной)
- Ответственный: трек «Конверсии и лидген»
- Дедлайн: 7 дней

**Связанные:**
- Гипотеза, которая родилась отсюда: solution «оплата картой на посадочной билет+проживание»

**Авторство:**
- Кто принял решение: команда трека
```
