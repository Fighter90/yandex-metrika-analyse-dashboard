# Skill — Decision Log (ProductCamp adaptation)

> Принцип: «Если решение не записано — его не было».
> Адаптировано из https://github.com/Voronik1801/Podlodka_crew_AI_Product/blob/main/skill-decision-log.md

---

## Назначение

Превращает каждый цикл проверки гипотезы в запись, которую агент «помнит» при следующем обращении. И команда тоже помнит — что важнее.

В контексте ProductCamp — фиксируем результат каждой проверки гипотез (problem и solution), чтобы:
- Не перепроверять то, что уже отбросили.
- Иметь доказательную базу для отчёта (DOCX/PDF секция Deliver).
- Передавать контекст между волонтёрами трека.

## Где хранить

В этом проекте — **сразу в трёх местах** (одна команда + автоматизация):

1. **Таблица `decisions` в SQLite** — основной source of truth, рендерится в дашборд и в отчёт.
2. **Markdown в `data/decisions/DL-{N}.md`** — экспорт для git-tracking и review через PR.
3. **Ссылка в `CLAUDE.md`** на 3 последних DL — чтобы агент при следующем запуске учитывал контекст.

Запись в БД → автогенерация .md → ссылка в CLAUDE.md обновляется через `pnpm new-decision`.

## Шаблон записи (.md и БД)

```markdown
## DL-{номер} — {короткая суть решения} ({дата})

**Связанная гипотеза:** #{id} «{title}»
**Phase:** {define | develop}
**Kind:** {problem | solution}

**Полная формулировка гипотезы:**
> «{subject} {action} {solution}, если {condition}»

**Что проверяли:**
- Метод: {synthetic / live / quantitative / market / mixed}
- Объём: {5 интервью / 12k сессий за 14 дней / 8 источников}
- Период проверки: {N дней}

**Что узнали (3-5 пунктов):**
1. {вывод}
2. {вывод}
3. {вывод}

**Цитаты / данные (доказательная база):**
> «{прямая цитата из CustDev}»
> Источник: {имя респондента, синтетический архетип / реальный}

> «{цифра из метрики}»
> Источник: raw_response #{id}, фильтр: {фильтр}, период: {период}

**Решение:**
- 🟢 Зелёный — продолжаем, внедряем (что именно)
- 🟡 Жёлтый — pivot: {что меняем в гипотезе}
- 🔴 Красный — стоп: {почему окончательно, что не стоит проверять повторно}

**Что делаем дальше:**
- Следующий шаг: {конкретное действие — A/B-тест / правка лендинга / следующая гипотеза}
- Ответственный: {имя из команды трека}
- Дедлайн: {дата}

**Связанные:**
- Предыдущее решение: DL-{N} (если pivot из него)
- Гипотезы, рождённые отсюда: #{id1}, #{id2}

**Авторство:**
- Кто принял решение: {Сергей / Лиза / совместно с командой}
- Кто участвовал в обсуждении: {список}
```

## Промпт для генерации черновика DL

```
На основе нашей сессии проверки гипотезы — заполни Decision Log по шаблону:

{вставь шаблон выше}

Контекст:
- Гипотеза: {вставь полную формулировку с допущениями и критериями}
- Метод проверки: {что делали}
- Полученные данные: {вставь цифры из Метрики и/или транскрипт CustDev}

Учти только то, что мы реально проверили в этой сессии.
НЕ выдумывай данные. Если чего-то не было — оставь поле пустым с пометкой "не проверяли".
НЕ интерпретируй цифры за пределами того, что они прямо показывают.

Особенно важно:
- Цитаты — точные, без перефраза. Если перефраз — пометь "[пересказ]".
- Решение (green/yellow/red) сопоставь с пред-заданными критериями гипотезы.
  Если данные не дотягивают ни до одного критерия — статус "inconclusive", не выдумывай.
- Следующий шаг — конкретный и с дедлайном. Не «изучить вопрос», а «провести 3 CustDev до пятницы».
```

## Что делает Decision Log сильным

- **Цитаты и цифры** — без них это «мне показалось».
- **Связь с предыдущими** — видно эволюцию мышления команды.
- **Авторство** — понятно, кто несёт ответственность.
- **Следующий шаг** — иначе решение зависает.
- **Привязка к гипотезе** — каждый DL ссылается на конкретный hypothesis_id в БД; статус гипотезы автоматически обновляется на outcome DL.

## Антипаттерны

- «Мы решили продолжать» без критериев — это не решение, а пауза.
- Список «выводов» без следующих действий — мёртвый документ.
- Один человек заполняет за всю команду — теряется контекст.
- DL без срока пересмотра — устаревает.
- DL без привязки к конкретному hypothesis_id — нельзя посчитать «процент проверенных гипотез».

## Бонус — замкнутая петля

После заполнения:

1. Запись в БД → автоматический экспорт в `data/decisions/DL-{N}.md`.
2. Скрипт `pnpm new-decision` обновляет раздел Decision Log в `CLAUDE.md` (3 последних).
3. Следующий раз при запуске Claude Code — агент учтёт прошлые DL и не предложит то, что уже отбросили.

В отчёте (DOCX/PDF) секция Deliver автоматически содержит все DL за период с полным форматом.
